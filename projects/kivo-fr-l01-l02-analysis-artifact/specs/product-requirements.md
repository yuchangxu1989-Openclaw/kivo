# KIVO FR-L01 + FR-L02 分析产物 - 产品需求规格

OpenClaw（sa-02 子Agent）/ 2026-05-24

---

## 1. 用户人群

KIVO 全部用户。重点画像：

- 关心系统抽取知识过程是否可解释、可审核的用户。
- 不愿意让系统直接把抽取结果写进知识库、要先看一眼候选实体和候选关联的用户。
- 维护多个领域知识库、需要在批量入库前对低置信度候选做人工把关的用户。
- 想看到"这次抽取从哪些原始素材里得到了什么候选信息"，避免知识库被噪声污染的用户。

---

## 2. 痛点

- 知识抽取目前只看到最终条目，看不到中间推理过程，用户没法判断这条知识的可信度。
- 候选实体、候选关联、潜在冲突这些中间结果没有沉淀，每次抽取过程结束就丢失，无法追溯。
- 低置信度的抽取结果直接进库会污染知识库，用户事后难以清理。
- 用户没有"我先看一眼候选清单、确认后再入库"的入口，要么全自动要么全手动，缺中间档位。
- 抽取过程中产生的 gap 候选和推荐调研问题被吞掉，无法接入调研管理形成正向循环。

---

## 3. 原始需求

让知识抽取过程拆成"分析 → 生成"两步：先把原始素材分析成结构化的候选清单（候选实体、候选关联、候选冲突、缺口建议、推荐调研问题），作为可持久化、可追溯的中间产物落库；再由审核机制（自动 / 人工）决定哪些候选项进入正式知识生成与入库。审核 UI 让用户能逐条确认或拒绝低置信度候选，通过审核的候选项以确定的接口进入正式知识库。

本 spec 只覆盖"分析产物生成 → 落库 → 审核 → 消费"这条主链路，不重新定义具体的知识条目结构（属 FR-A 系列）、不重新定义入库后的去重归域算法（属 FR-P 系列）。

---

## 4. 用户体验流

1. 用户提交原始素材（文档 / 对话片段 / 调研报告）触发知识抽取，抽取流程不再直接产出知识条目，而是先产出一份分析产物，落库为可追溯的中间结果。
2. 分析产物完成后，用户在工作台的"分析产物"页面看到本次产出的候选清单，包括候选实体、候选关联、潜在冲突、缺口建议和推荐调研问题。
3. 高置信度产物默认走自动通道，候选项直接进入知识生成与入库，用户在审计列表里能事后看到"这一批是自动消费的"。
4. 低置信度产物自动进入"待审核"队列，用户打开队列后看到每条候选的内容、置信度、来源引用，能逐条点确认 / 拒绝 / 编辑。
5. 用户确认通过的候选项进入知识生成与入库阶段，拒绝的候选项被标记为"已拒绝"留在分析产物里，不进库但不丢失记录。
6. 分析产物里的缺口建议和推荐调研问题以"建议清单"的形式显示，用户可一键把建议作为新的调研任务发起，把分析产物和调研管理打通。
7. 任何时候用户都可以打开历史分析产物，看到"这条知识是从哪份分析产物里出来的"，完成端到端溯源。

---

## 5. 功能需求

### FR-A 分析产物生成

知识抽取的第一步必须产出结构化分析产物，而非直接产出知识条目。

- AC1：每次抽取流程都生成一份分析产物记录，至少包含：来源引用（原始素材 ID / 对话 ID / 报告 ID）、提取的断言列表、候选实体、候选概念、候选关联、候选冲突、缺口建议、推荐调研问题、需人工审核事项、置信度评分。
- AC2：分析产物里每个候选项都自带置信度评分，评分粒度到单条候选，不允许整份产物只有一个总分。
- AC3：分析产物的候选项必须能被反向追溯到来源素材的具体片段（文档段落 / 对话轮次），用于后续审计与追责。
- AC4：分析产物生成必须使用 LLM 语义识别，禁止用关键词匹配、正则、模板冒充语义抽取。
- AC5：抽取流程中如果某段素材生成失败，对应分析产物记录该段为"生成失败"，但已经成功的部分仍然落库，不允许整份产物因部分失败而丢弃。

### FR-B 分析产物落库

分析产物作为长期可追溯的中间数据持久化，不随知识条目生成完即丢弃。

- AC1：分析产物作为系统专属的持久化记录存储，字段覆盖 FR-A AC1 中描述的全部结构化字段。
- AC2：每条分析产物记录拥有唯一 ID，并与触发抽取的来源素材建立外键 / 引用关系，支持按来源、时间、领域库多维查询。
- AC3：分析产物落库后不允许被原地静默覆盖，新一次抽取产生新的分析产物记录，旧记录保留用于历史溯源。
- AC4：分析产物必须支持持久化保留至少 N 天（默认 90 天，可配置），保留期内可被审核 UI 和消费 API 直接读取。
- AC5：分析产物的写入是事务化的，要么整份成功落库、要么完全不写，不允许出现"产物记录建了但候选清单为空"的中间态。

### FR-C 分析产物审核 UI

工作台提供分析产物浏览和审核界面，用户可对低置信度候选逐条决策。

- AC1：工作台提供"分析产物"页面，列表展示所有分析产物记录，按时间倒序排列，列表至少展示：产出时间、来源摘要、候选数、整体状态（待审核 / 部分审核 / 已通过 / 已拒绝 / 已自动消费）。
- AC2：低置信度的分析产物（整份或单条候选置信度低于阈值）自动进入"待审核"队列，列表支持按状态筛选；未审核的低置信度候选项不允许进入消费接口，高置信度产物的自动消费在审计日志里可见、不要求用户点确认。
- AC3：用户点击进入产物详情页可看到候选清单全文，每条候选支持逐条"确认 / 拒绝 / 编辑后确认"三种操作，操作必须留下操作人和时间。
- AC4：详情页同时展示候选项与原素材片段的对照关系，用户能直接看到"这条候选是从哪段原文里抽出来的"。
- AC5：审核 UI 上显示分析产物中的缺口建议和推荐调研问题，每条建议旁出现"作为调研任务发起"的按钮，点击后接入 FR-D01 的调研发起入口。

### FR-D 消费 API：审核通过候选进入正式知识库

通过审核的候选项以明确定义的接口进入正式知识生成与入库流程。

- AC1：分析产物提供消费接口，输入是产物 ID + 候选项 ID 列表，输出是写入正式知识库的条目 ID 列表，整个过程必须经过主 spec 已定义的统一质量门禁。
- AC2：高置信度的分析产物在落库后自动调用消费接口，把候选项推进到正式入库；低置信度的产物只有在用户审核通过后才能调用消费接口。
- AC3：消费接口是幂等的：同一个产物 ID + 候选项 ID 组合多次调用，最终知识库里只存在一份对应条目，避免重复入库。
- AC4：候选项被消费后状态标记为"已消费"，分析产物记录上保留消费时间、目标领域库、对应的知识条目 ID，形成端到端可追溯链路。
- AC5：被用户拒绝的候选项不允许进入消费接口，状态标记为"已拒绝"留在分析产物里，但不写入正式知识库；如用户后续改主意，需要重新触发审核流程。
