# KIVO FR-D01 + FR-D02 调研链路 - 产品需求规格

OpenClaw（sa-02 子Agent）/ 2026-05-24

---

## 1. 用户人群

KIVO 全部用户。重点画像：

- 在某个领域库里持续提问、希望系统帮自己持续补盲的用户。
- 习惯用对话发起需求、不愿意去专门的"任务管理页"配置参数的用户。
- 同时维护多个领域知识库，需要看到每个领域当前在调研什么、调研到哪一步的用户。
- 关心调研产出能否真正变成长期沉淀，而不是一次性回答的用户。
- 把 KIVO 当成长期知识助手、需要看到调研历史与失败记录的用户。
- 同时调研多条问题、需要看到队列负载、不想被重复任务干扰的用户。
- 希望调研报告能在领域库中隔离存储、不被跨领域瞱到另一个库里的用户。

---

## 2. 痛点

- 用户在领域库提问时，系统只给一次性回答，没有"我先帮你调研一下、出报告再回来"的持续动作，复杂问题永远停在表层。
- 调研被当成一次性对话流程，没有任务实体，用户离开会话后就追不到后续，只能凭印象或者去翻聊天记录。
- 即使有报告产出，用户也不知道哪些已经做完、哪些还在跑、哪些失败了，调研工作变成黑盒。
- 调研做完之后报告散落在对话里，用户没法集中查看一个领域历史上调研过哪些问题、得到过哪些结论。
- 调研报告和领域知识库是两套断开的系统，用户读完报告还要手动决定是否入库、怎么入库，断点摩擦大。
- 失败任务被静默吞掉，用户不知道是什么原因失败、能不能重跑，导致重要问题永远没产出。
- 同一个问题被反复发起调研，没有合并机制，队列被重复任务占满，真正新的问题排不上。
- 自动入库会把质量参差的调研报告全部丢进知识库，用户没有最后把关的机会，知识库容易被噪声污染。
- 跨领域库使用时，如果调研任务归属错乱，用户在 A 库的问题报告进了 B 库，回头很难找到，知识沉淀就乱了。

---

## 3. 原始需求

让用户在某个领域库里通过对话提问时，系统自动判断这个问题适合作为调研任务，建立一条独立的调研记录、进入队列，由后台执行单元异步跑出报告，并在领域库的调研队列页面持续可见。报告完成后由用户在页面上做"采纳 / 丢弃"决策，采纳后接 FR-P05 的 Wiki 准入流程把报告内容沉淀进领域知识库。失败任务必须显式可见、可重跑，不允许被静默吞掉。

本 spec 只覆盖"对话发起 → 任务建档 → 队列执行 → 报告产出 → 用户决策 → 接入 P05"这条主链路，不重新定义报告内部的拆解去重归域算法（属 FR-P05）、不重新定义领域匹配逻辑（属 FR-P04）、不重新定义意图识别细节（属 FR-E03/E04）。

---

## 4. 用户体验流

1. 用户在某个领域库的对话入口里提问，比如"帮我调研一下 XXX 在 2025 年的最新进展"，对话窗内立即得到"已为你建立调研任务，进入队列"的反馈，不阻塞用户继续聊。
2. 系统在后台为这次提问建立一条调研任务记录，绑定发起人、所属领域库、原始问题、创建时间，初始状态进入"待调研"，并把这条任务挂到当前对话的可追溯线索上。
3. 调研队列页面在该领域库下出现这条新任务，用户随时可以切到队列页查看队列长度、队头任务、当前在跑的任务，离开队列页再回来不丢失任何任务。
4. 系统在后台按节奏扫描队列，挑出最早的待调研任务交由后台执行单元处理，任务状态变为"进行中"，队列页上的状态自动刷新，用户不需要手动刷新页面就能看到任务推进。
5. 后台执行单元跑完后产出一份调研报告，任务状态变为"已完成"，队列列表里这条记录出现"查看报告"和"采纳 / 丢弃"两个动作，用户能在队列页内直接读完报告再决策。
6. 用户在队列页点开报告预览，决定保留就点采纳，触发 FR-P05 准入流程把报告拆解入库；觉得无价值就点丢弃，任务状态变为"已丢弃"，不进知识库但记录保留。
7. 执行失败的任务状态被标记为"失败"，队列页展示失败原因摘要，用户可决定让系统重跑或丢弃，不会被默默吞掉；多次重跑仍失败的任务给出明确提示，避免无限重试浪费资源。
8. 用户后续再问类似问题时，系统识别到队列里已有相近的待调研或已完成任务，直接复用历史结果或合并到已有任务，不会反复占用队列资源。
9. 用户在多个领域库之间切换查看队列时，每个领域库只展示自己的调研任务，互不干扰，发起调研也只能在当前领域库内生效，避免任务归属错乱。
10. 用户首次进入空队列时看到清晰的引导文案，告知"调研由对话发起，不在 Web 创建"，避免用户在页面上找不到操作入口而困惑。

## 5. 功能需求

### FR-A 对话发起调研任务

用户在领域库对话中提出可作为调研对象的问题时，系统自动建立调研任务，不需要用户离开对话去填表单。

- AC1：用户在某个领域库的对话上下文中提出调研类请求时，系统通过语义判断识别"这是一次需要调研的请求"，自动建立调研任务记录，对话内即时回执"已建任务、进入队列"，且对话主线程不被阻塞。
- AC2：调研任务记录至少包含：任务 ID、发起人、所属领域库、原始问题、创建时间、初始状态（待调研）、来源对话引用，所有字段必须落库，不允许只在内存中流转。
- AC3：识别为调研任务的判断必须基于 LLM 语义理解，禁止用关键词匹配、正则、模板切句冒充意图识别；置信度低于阈值时不自动建任务，对话里明确提示"如需调研请显式说'帮我调研 XXX'"。
- AC4：同一领域库内、同一发起人、语义高度相似的待调研任务（向量相似度超过阈值）被合并去重；命中已完成任务时直接给出历史报告引用，不重复建新任务。
- AC5：调研任务的发起范围严格绑定当前对话所在的领域库，跨领域库的调研请求要求用户显式切换领域库或在对话中指明，避免任务归属错乱。

### FR-B 调研队列后台执行

调研任务由后台持续扫描队列、交由执行单元异步跑出报告，整个过程不依赖用户停留在页面。

- AC1：后台按可配置节奏（默认每 N 分钟）扫描调研任务，挑出"待调研"状态的最早任务交给后台执行单元，任务状态变为"进行中"；单次扫描的并发上限可配置，后台执行尊重资源边界。
- AC2：后台执行单元跑完产出报告后，调研任务记录中写入：完成时间、报告产出引用（报告文件 / 飞书文档链接）、状态变为"已完成"，写入是事务化的，避免出现"标记完成但报告引用为空"的中间态。
- AC3：执行失败时任务状态标记为"失败"，必须记录失败原因摘要（错误类别 + 简短描述），队列页可见；失败任务支持用户在队列页发起重跑，重跑时复用原任务记录而非新建任务。
- AC4：后台执行要有幂等保护：同一任务不能被两个扫描周期同时派发，已经在跑的任务必须先完成或失败才能重新进入调度；后台执行全局开关支持暂停，暂停期间已在跑的任务允许跑完、新任务不再派发。
- AC5：后台执行在每个扫描周期内必须记录运行日志（扫描时间、本轮派发任务数、失败任务数），日志可在系统操作日志页面查看，作为运行态可观测性的兜底。

### FR-C 调研队列页面（Web）

每个领域库下都有自己的调研队列页面，用户在这里集中看到该领域历史上的所有调研任务、当前状态、报告内容和决策动作。

- AC1：领域库导航中提供"调研队列"入口，进入后展示该领域库下的全部调研任务，按创建时间倒序排列；列表每条记录展示：原始问题、状态、创建时间、完成时间（若有）、操作区。
- AC2：列表支持按状态筛选，至少包含"全部 / 待调研 / 进行中 / 已完成 / 失败 / 已采纳 / 已丢弃"；顶部展示该领域库的队列概览（待调研数 / 进行中数 / 已完成数 / 失败数），方便用户快速判断当前调研负载。
- AC3：已完成状态的记录在操作区出现"查看报告 / 采纳 / 丢弃"三个按钮，点击"查看报告"在页面内弹出报告预览不强制跳出；失败状态的记录在操作区出现"重跑 / 丢弃"按钮；其他状态不展示这两组按钮避免误操作。
- AC4：页面纯查看 + 决策，不提供"在 Web 创建新调研任务"的入口，调研发起入口唯一来源是对话；列表分页与排序参数对前端可见，每页默认 N 条（可配置），避免任务积压时一次性加载导致页面卡顿。
- AC5：所有状态变更（包括后台执行触发的状态切换）必须在用户停留页面时及时刷新到列表上，刷新方式可以是轮询或推送，但延迟不允许超过可配置的可见性阈值。

### FR-D 报告采纳接入 P05 准入流程

用户在队列页点击"采纳"是 Wiki 准入的唯一触发点，采纳之后由 FR-P05 接管后续拆解、去重、归域、入库。

- AC1：队列页"采纳"按钮被点击时，系统将该调研任务对应的报告作为输入，调用 FR-P05 定义的 Wiki 准入入口，并把任务 ID、所属领域库、报告引用一同传入。
- AC2：采纳触发后调研任务状态立即变为"采纳处理中"，按钮变灰禁止重复点击；FR-P05 入库流程完成后状态最终落到"已采纳"，并记录入库结果摘要（写入条目数、跳过条目数、目标领域）。
- AC3：用户点击"丢弃"时任务状态变为"已丢弃"，不调用 P05 流程、不写入任何知识条目，但记录保留在队列页用于审计追溯。
- AC4：采纳是用户专属动作，调研任务在后台执行完成后默认停留在"已完成"状态等用户决策，禁止系统内部任何后台机制绕过用户直接触发 P05 入库。
- AC5：采纳与丢弃动作均需在调研任务记录上留下操作日志（操作人、操作时间、动作类型），作为审计与追溯线索。
