# KIVO 走查 P1 修复 - 搜索相关性 + 上传状态反馈

OpenClaw（pm-02 子Agent）/ 2026-05-24

## 1. 用户人群

- **陌生小白用户**：第一次打开 KIVO Web，没有看过任何文档，按浏览器里的按钮直觉操作。判断 KIVO 是否能用，靠的是搜索结果是否对题、操作有没有反馈。
- **知识工作者**：日常用 KIVO 沉淀学科资料、查找已收录知识，关心搜索能不能精准命中专业内容，新材料上传后能不能看到处理进度。
- **浏览器场景**：所有交互发生在 KIVO Web 端（白底黑字），用户只看页面，不看终端日志，不读 DB 表。

## 2. 痛点

走查实证（reports/kivo-stranger-walkthrough-2026-05-24.md）：

- 知识搜索页搜「概率论」「贝叶斯」「全概率公式」三次，每次都返回 60 条结果，前排却是「能力复用优先铁律」「知识准入的三重测试」「多层级记忆系统」等 KIVO 自身治理规则，而不是学科内容。陌生用户会判断「搜索坏了」。
- 原始资料库点「导入素材」上传 PDF 后，弹窗关闭，30 秒后刷新仍显示「共 8 条素材」，没有新卡片，也没有「上传中 / 解析中 / 编译中 / 失败」任何状态文字。用户视角是「提交后消失」，不知道成功失败。
- 领域 wiki 内部搜索结果卡片标着「关键词匹配」，与产品页面承诺的「语义检索」直接冲突，让用户怀疑搜索能力是假的。

根因层面：搜索池子混合了系统治理类条目和学科知识，且排序权重让治理条目压过学科条目；上传链路前端走的是旧 `/api/materials/ingest`，提交后没有任何状态轮询和列表刷新。

## 3. 原始需求

- 搜索得对：搜什么词，前排返回的就是相关学科/领域知识，治理类系统条目默认不出现在搜索池里。
- 上传得有反馈：提交瞬间用户就知道材料已被收下，列表立即出现新卡片，状态文案随处理进度自动更新，处理失败要写明原因。

## 4. 用户体验流

### 4.1 搜索

1. 用户进入「知识搜索」页，输入查询词（学科名 / 概念 / 公式名）。
2. 系统按向量相似度返回结果，前排全是相关学科/领域知识条目。
3. 默认搜索范围不包含 KIVO 自身的治理规则、元规则、系统配置说明。
4. 想看系统规则的用户，可以打开搜索框旁的「包含系统规则」开关，再次搜索时治理类条目才会进入候选池。
5. 任何结果卡片不出现「关键词匹配」标签；命中标签必须如实反映检索方式（语义检索 / 向量召回）。

### 4.2 上传

1. 用户在原始资料库点「导入素材」，选文件、填标题，点提交。
2. 提交瞬间出现 toast「已登记，编译中」，弹窗关闭。
3. 列表顶部立即出现新卡片，状态显示 `pending`。
4. 卡片自身每 3-5 秒向 status 接口轮询一次，状态依次推进：`pending` → `slicing` → `extracting` → `injecting` → `done`。
5. 任一阶段失败 → 状态显示 `failed`，卡片上写明失败原因（解析失败 / 抽取失败 / 注入失败 + 简短原因句）。
6. `done` 后停止轮询，卡片显示完成态，可点开查看抽取出的知识条目。

## 5. 功能需求

### FR-A：搜索相关性修复

搜索池范围、排序口径、UI 标签三处一起对齐到「向量语义检索」的承诺。

#### AC

- **AC1**：知识搜索结果按向量相似度排序，禁止用 sqlite FTS5、关键词匹配、正则匹配冒充语义检索。任何 fallback 也不允许走非语义路径，向量服务不可用时直接报错并提示用户，而不是降级到关键词。
- **AC2**：默认搜索范围排除 governance / meta / system 类条目（KIVO 自身的运行规则、提示词、治理说明）。这些条目和学科知识不应共处一个搜索池。前端在搜索框旁提供「包含系统规则」开关，开启后才把这些条目加入候选池。开关状态在会话内保持，刷新后回默认（关闭）。
- **AC3**：实测搜领域名（用户提到的学科占位词，如概率论 / 高数 / 任意已收录领域）时，前 10 条搜索结果全部为该领域内的学科知识条目，治理规则数量为 0。

### FR-B：上传状态反馈

把 ImportMaterialButton 链路切到 `/api/v1/wiki/upload`，前端立即写入新卡片并自动轮询状态。

#### AC

- **AC1**：用户点提交后，前端立即弹出 toast「已登记，编译中」并关闭上传弹窗，原始资料库列表顶部立即出现一张新卡片（标题 = 用户填写的标题，状态 = `pending`，时间戳 = 当前时间）。
- **AC2**：新卡片自动调用 status 接口（`/api/v1/wiki/upload/{id}/status` 或同等链路）轮询，每 3-5 秒一次，直到状态进入 `done` 或 `failed` 后停止轮询。
- **AC3**：状态文案使用统一枚举：`pending`（已登记）、`slicing`（切片中）、`extracting`（抽取知识中）、`injecting`（写入图谱中）、`done`（已完成）、`failed`（处理失败）。文案对用户可读，不是英文 raw 值。
- **AC4**：状态为 `failed` 时，卡片上显示失败原因短句（如「PDF 解析失败：文件损坏」「LLM 抽取超时」「注入图谱失败」），原因来自后端返回的 `error.message` 字段。
- **AC5**：ImportMaterialButton 切换到新链路 `/api/v1/wiki/upload`，旧链路 `/api/materials/ingest` 在前端不再调用，并在代码层删除前端引用，避免回退。新旧链路的字段差异由前端适配层吸收。

## 6. 非功能需求

- 搜索结果首屏延迟 ≤ 1.5s（向量检索 + 排序合计）。
- 上传 toast 与新卡片渲染必须 ≤ 500ms 内可见，不依赖后端确认。
- 状态轮询接口 P95 响应 ≤ 200ms。
- 视觉硬约束：白底黑字，禁止深色主题、霓虹紫光、黑底配色。
- 通用化：搜索池过滤策略、状态枚举文案、轮询间隔均通过配置可调，不硬编码学科名或具体路径。

## 7. 验收方式

- 陌生用户首次打开 KIVO Web，依次完成「搜任意已收录领域 → 看到 10 条相关知识」「上传一个 PDF → 看到 toast → 看到新卡片 → 状态自动从 pending 推进到 done」整套流程，全程不依赖命令行、不读日志。
- 走查回归：在 reports/ 下产出新一轮陌生人走查报告，截图覆盖搜索 top10 + 上传卡片状态推进。

## 8. 范围与边界

- 本 spec 只覆盖知识搜索页 + 原始资料库上传链路两处。
- 不动领域 wiki 详情页、仪表盘 Default Space 链接（属于另一份 P1 spec）。
- 不动向量服务的部署形态，假定 bge-m3 已在线；若不在线，由运行态告警处理，不在本次范围。
- 治理类条目的判定方式（按知识类型 governance/meta/system 还是按 source path）由实施时基于现有 schema 选择，但默认行为必须满足 AC2。
