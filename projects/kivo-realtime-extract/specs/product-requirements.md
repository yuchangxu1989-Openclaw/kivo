# KIVO 实时知识提取 Hook 集成 — 产品需求规格

OpenClaw（pm-01 子Agent）| 2025-05-22

## 问题陈述

KIVO 当前仅有 cron 批量提取（每 2 小时），用户纠偏、决策、新规则最多延迟 2 小时才能进入知识库。需要在 message:received 事件中增加实时提取，让高价值知识秒级入库。

## 现有资产

- `realtime-capture.ts`（133 行）：提取逻辑已实现，未接入事件流
- `kivo-intent-injection/handler.js`：message:received 时注入知识（只读不写）
- 批量提取 cron：每 2 小时全量扫描历史会话

## 功能需求

### FR-FIX-01：message:received 实时提取集成

| AC 编号 | 验收条件 |
|---------|----------|
| AC-01 | 在 kivo-intent-injection handler 的 message:received 事件处理中，增加异步提取调用 |
| AC-02 | 复用现有 realtime-capture.ts 模块，不重复实现提取逻辑 |
| AC-03 | 提取结果经质量门禁评分后写入 DB（复用现有门禁规则） |
| AC-04 | 提取为异步 fire-and-forget，不阻塞消息响应链路 |
| AC-05 | 提取失败不影响注入流程，错误仅记录日志（console.error + 结构化字段） |
| AC-06 | 消息长度 <50 字符时跳过提取，避免短消息噪音 |

### FR-FIX-02：提取去重

| AC 编号 | 验收条件 |
|---------|----------|
| AC-07 | 实时提取的知识点写入前，与已有条目做语义相似度比对（复用 BGE-M3 向量） |
| AC-08 | 相似度 >0.92 视为重复，不写入 DB |
| AC-09 | 去重环节异常（向量服务不可用等）时降级为直接写入，宁可重复不可丢失 |

## 非功能约束

- 实时提取延迟对消息响应的影响：0ms（异步，不在关键路径）
- 单次提取超时上限：10s（超时则放弃本次，记日志）
- 与批量 cron 共存：cron 继续运行，作为兜底；实时提取覆盖的条目 cron 自然去重跳过
- 日志级别：正常提取 debug，跳过/去重 info，失败 error

## 不做

- 不改变现有注入逻辑（只读路径不变）
- 不替代 cron 批量提取（cron 作为兜底保留）
- 不新增 API 端点或 Web 界面
