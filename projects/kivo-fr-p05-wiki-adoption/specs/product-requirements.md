# KIVO FR-P05 Wiki 准入流程 - 产品需求规格

OpenClaw（sa-01 子Agent）/ 2026-05-24

---

## 1. 用户人群

KIVO 全部用户。重点画像：

- 已经习惯通过对话发起调研、由后台执行单元出报告的用户。
- 把调研报告当一手资料、希望沉淀成长期知识库的用户。
- 同时维护多个领域知识库（LLM Wiki），需要把不同主题的报告归到正确领域的用户。

---

## 2. 痛点

- 调研做完后只剩一份 md / 飞书文档，没法变成可检索、可关联的领域知识。
- 调研系统和 Wiki 系统目前是断开的：报告完成 → 用户读完就完了，知识没有进库。
- 没有"我看完觉得这份值得收"的明确动作，导致系统自动入库容易把质量差的报告也丢进 Wiki。
- 同一个主题的内容反复被调研，不知道前几次的成果是否已经入库，库里出现大量语义重复。
- 报告里同时讲多个知识点，整篇丢进库会让条目颗粒度过粗，后续检索命中差。

---

## 3. 原始需求

让用户在调研管理页面对已完成的调研报告做一次显式"采纳"动作之后，系统自动把这份报告拆解成原子级领域知识条目，去重、归域、入库，全过程不需要用户再操作。入库权始终在用户手里，系统不能自动触发。

本 spec 只覆盖"采纳触发 → 拆解 → 去重 → 入库 → 归域"这条主链路，不重新定义 Wiki 页面聚合（FR-P06）、调研页面交互（FR-W07）、领域匹配算法（FR-P04）。

---

## 4. 用户体验流

1. 用户在调研管理页面看到一份已完成的调研报告，阅读后判断值得入库。
2. 用户点击"采纳"按钮，系统给出确认弹窗，提示"采纳后报告将被自动拆解并写入领域知识库"。
3. 用户确认后，按钮立即进入"采纳处理中"状态，调研记录上展示一条入库进度。
4. 系统在后台拆解报告、做向量去重、归域、写入领域知识库，整个过程异步执行，不阻塞用户页面。
5. 用户能在同一张报告卡片上看到入库结果概览：写入条目数、跳过的重复条目数、归到哪个领域库。
6. 入库完成后，报告状态从"已完成"变为"已采纳"，重复点击采纳按钮被禁用，避免二次入库。
7. 用户随后到对应领域知识库页面，可以看到从这份报告产生的新条目，每条都能溯源回原报告。

---

## 5. 功能需求

### FR-A 调研报告采纳按钮

在调研管理页面（FR-W07）上提供显式的"采纳"入口，作为 Wiki 准入的唯一触发点。

- AC1：调研管理页面每条已完成报告的操作区出现"采纳"按钮，未完成 / 已采纳 / 失败状态的报告不展示该按钮。
- AC2：点击"采纳"弹出确认框，明确告知后续会自动拆解入库，确认后才真正触发，否则不做任何写入。
- AC3：确认后按钮变为不可点击的"采纳处理中"状态，配合一条入库进度显示，禁止用户在同一报告上重复触发。
- AC4：采纳是用户专属动作，后台执行单元与周期调度不允许调用同一入口，后端入库 API 必须拒绝非用户主体的调用。
- AC5：采纳后报告卡片永久标注采纳时间与入库结果摘要，作为审计与追溯线索。

### FR-B LLM 拆解为知识条目

把整份调研报告拆解成原子级领域知识条目，颗粒度对齐主 spec 的提取约束。

- AC1：拆解流程复用既有的 Wiki 页面编译能力，按主 spec 中"原子级知识"的定义产出多条独立条目，不允许整篇报告作为一条条目。
- AC2：每条产出的条目包含完整正文、知识类型、所属主题、与原报告的溯源信息（报告 ID + 段落定位）。
- AC3：拆解过程必须使用 LLM 语义判断，禁止用关键词匹配、正则、模板切段冒充语义拆解。
- AC4：拆解失败（LLM 报错、内容不可解析）时整份报告入库流程中止，状态回退为已完成，让用户看到失败原因，不允许半入库。
- AC5：拆解产出在写入前先经过主 spec 已定义的统一质量门禁，未达标条目直接丢弃并记入日志。

### FR-C 向量去重检测

在写入领域知识库前对每条候选条目做语义去重，防止 Wiki 出现重复内容。

- AC1：每条候选条目都会与目标领域知识库已有条目做向量相似度比对，相似度达到 spec 全局阈值的视为重复。
- AC2：判定为重复的条目默认丢弃并记日志（重复来源 + 已有条目 ID），不静默覆盖已有条目。
- AC3：去重必须基于向量语义，禁止退回到关键词匹配或字面比对。
- AC4：候选条目内部也要互相去重，避免同一份报告里语义相同的多条片段被同时写入。
- AC5：去重结果作为采纳摘要的一部分回显给用户（写入 N 条 / 跳过 M 条 / 因质量门禁丢弃 K 条）。

### FR-D 自动入库与归域

去重通过的条目自动写入领域知识库，并归属到语义最匹配的领域。

- AC1：每条候选条目复用主 spec 中的领域匹配逻辑，自动归到语义最匹配的领域知识库，不需要用户手动选择。
- AC2：匹配置信度低于阈值的条目归入默认未分类领域，等用户后续在 Wiki 页面手动调整，不允许阻塞整批入库。
- AC3：写入操作必须是事务化的：要么整批成功，要么失败时回滚已写入条目，不留半截数据。
- AC4：入库结果回写到调研报告记录上：采纳时间、目标领域、写入条目数、跳过条目数、失败条目数。
- AC5：同一份报告的二次"采纳"被入库幂等机制拦截，无论前端是否禁用按钮，后端都不会写出重复条目。
