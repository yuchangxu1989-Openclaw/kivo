# KIVO 终局修复 - 产品需求规格

OpenClaw（pm-01 子Agent）| 2026-05-22

---

## 概述

本 spec 针对 KIVO 差距扫描发现的 P0+P1 缺口，定义每个缺口的修复验收标准。所有 FR 引用原始 product-requirements.md 中的编号，修复范围限定为已有功能的补全，不新增功能。

## P0 缺口（必须修复）

### FIX-01：FR-A06 实时知识提取（补全 message 级触发）

**现状**：仅有 cron 批量提取（每 2 小时），用户纠偏/决策/新规则最多等 2 小时才能进入知识库。`kivo-intent-injection` handler 只读不写。

**修复目标**：在 `message:received` 事件触发时实时提取高价值知识并写入知识库。

**验收标准：**

- AC1：`hooks/kivo-intent-injection/handler.js` 在处理 `message:received` 事件时，除执行现有的知识注入（读）逻辑外，新增实时提取（写）逻辑。提取逻辑与注入逻辑解耦，提取失败不影响注入。
- AC2：实时提取调用 LLM 分析当前轮次的用户消息，识别高价值知识类型（纠偏、决策、新规则、事实更新、经验教训）。禁止关键词匹配或规则引擎，必须 LLM 语义判断。
- AC3：提取的知识条目经统一质量门禁（FR-N05）过滤后写入 `kivo.db` entries 表。门禁不通过的条目丢弃并记录日志。
- AC4：提取流程异步执行（不阻塞 handler 对消息的响应），使用 `setImmediate` 或等效机制将提取逻辑推入下一个事件循环。
- AC5：每条提取的知识条目包含来源追溯：`session_id`、`message_id`、时间戳，写入 entry 的 metadata。
- AC6：禁止基于消息字数/字符数/消息条数的预过滤。每条用户消息到达即触发提取流程，由 LLM 判断是否包含高价值知识。唯一允许的预过滤：纯系统生成内容（heartbeat、system prompt injection 等）。
- AC7：提取 prompt 中包含 FR-A07 定义的准入判定规则（三重测试）和原子粒度约束，从源头控制产出质量。
- AC8：实时提取与 cron 批量提取（FR-A05）通过 `message_id` 去重——已被实时提取处理过的消息，cron 批量提取时跳过。
- AC9：handler 中新增的提取逻辑有独立的 try-catch，任何异常只写错误日志，不抛出到 handler 主流程。

### FIX-02：领域知识库 Web 体验（FR-P01/P02 补全）

**现状**：`space-manager.ts`（42 行）和 `directory-manager.ts`（35 行）为骨架代码，Wiki 页面缺少完整的 CRUD 交互、条目浏览和搜索体验。

**修复目标**：领域知识库（LLM Wiki）页面具备完整的 space 管理、directory 管理、条目浏览和语义搜索能力。

**验收标准：**

- AC1：Space 管理页面支持创建、编辑、删除领域知识库。创建表单包含名称、描述、知识范围定义。名称全局唯一校验，重复时前端提示。
- AC2：Space 列表页展示每个库的名称、描述、条目数量、最近更新时间。支持按名称搜索过滤。
- AC3：Space 详情页（即 directory 视图）展示该库下所有知识条目，支持分页（每页 20 条）、按知识类型筛选、语义搜索（调用后端向量检索 API）。
- AC4：Directory 管理支持在 space 内创建、重命名、删除子目录。条目可在目录间拖拽移动或通过操作菜单移动。
- AC5：知识条目支持在 Wiki 页面内直接创建、编辑（标题+内容+类型+标签）、删除。删除前弹出确认对话框。
- AC6：条目详情页展示完整内容、元数据（类型、来源、创建时间、更新时间）、关联图谱节点链接。
- AC7：语义搜索输入框位于 space 详情页顶部，输入后实时调用向量检索 API 返回结果，结果高亮匹配条目。禁止前端关键词过滤冒充语义搜索。
- AC8：`space-manager.ts` 重写为完整的 React 组件，包含状态管理、API 调用、错误处理、加载态。最终代码量应反映完整 CRUD 逻辑（预估 200-400 行）。
- AC9：`directory-manager.ts` 重写为完整的树形目录组件，支持展开/折叠、右键菜单（新建/重命名/删除）、拖拽排序。最终代码量应反映完整交互逻辑（预估 150-300 行）。
- AC10：所有操作调用后端 API，操作成功后刷新列表，失败时展示错误提示（toast）。
- AC11：页面遵循 KIVO 全站浅色主题（白色背景 + 黑色文字），交互元素满足 WCAG AA 对比度标准。
- AC12：目录与条目列表必须有层级关系——目录是章节/分类（如"随机事件与概率"），条目是该分类下的具体知识点（如"条件概率公式""贝叶斯公式"）。禁止目录=条目同层级（当前 bug：目录和条目列表展示完全相同的数据）。
- AC13：知识条目内容支持 LaTeX 公式渲染（数学公式是领域知识的核心载体）。
- AC14：知识条目支持图片嵌入（图表、示意图等）。
- AC15：从 PDF 导入时必须提取结构化内容——定义、公式（LaTeX）、定理、典型例题，不能只提取标题+一句话摘要。
- AC16：每个知识点保留 PDF 来源引用（页码/段落位置），用户可点击追溯原文。

## P1 缺口（功能不完整）

### FIX-03：FR-E03 意图信号检测（扩展信号类型覆盖面）

**现状**：`signal-detector.ts`（71 行）过于精简，信号类型覆盖不足。

**修复目标**：信号检测覆盖 spec 定义的全部信号类型，检测 prompt 具备足够的语义区分能力。

**验收标准：**

- AC1：信号检测覆盖以下类型：纠偏（correction）、强调（emphasis）、声明（declaration）、规则（rule）、偏好（preference）、决策（decision）、约束（constraint）、经验教训（lesson_learned）、事实更新（fact_update）。每种类型在检测 prompt 中有明确的定义和示例。
- AC2：检测 prompt 对每种信号类型提供 2-3 个正例和 1-2 个负例，帮助 LLM 区分边界情况。
- AC3：支持 L2 自定义信号类型——用户可通过配置注册新信号类型（类型名、描述、检测 prompt 片段），自定义类型走同一条提取管线。
- AC4：每个检测到的信号包含：信号类型、置信度评分（0-1）、原始对话片段引用、检测理由（一句话说明为什么判定为该类型）。
- AC5：`signal-detector.ts` 重构后代码结构清晰：信号类型注册表、检测 prompt 构建器、结果解析器分离。最终代码量应反映完整逻辑（预估 200-350 行）。
- AC6：单 Agent 场景兼容——当环境只有一个 Agent 时，从主会话对话中收集信号，不依赖子 Agent 事件。

### FIX-04：FR-G04 图谱自动对齐（补全对齐检测逻辑）

**现状**：缺少独立的对齐检测逻辑，无法发现图谱与知识库的不一致。

**修复目标**：cron 任务能检测并修复 graph_nodes/graph_edges 与 entries 表的不一致。

**验收标准：**

- AC1：对齐检测逻辑作为独立模块实现（如 `graph-alignment-checker.ts`），不混入图谱构建逻辑。
- AC2：检测以下不一致类型：
  - 孤儿节点：graph_nodes 中存在但 entries 表中已删除的条目
  - 缺失节点：entries 表中存在但 graph_nodes 中没有的条目
  - 过期边：graph_edges 引用的节点已不存在
  - 权重漂移：entry 内容已更新但 edge weight 未重算
- AC3：检测到不一致后自动修复：删除孤儿节点和过期边、为缺失节点创建节点、对权重漂移的边触发重算。
- AC4：对齐检测由 cron 触发，调度时间排在图谱增量构建之后。频率可配置，默认每天一次。
- AC5：执行结果写入日志：检测到的不一致数量（按类型分类）、修复操作数量、执行耗时。
- AC6：CLI 命令 `kivo graph align` 可手动触发一次对齐检测+修复，支持 `--dry-run` 只检测不修复。

### FIX-05：FR-G05 注入图谱关联增强（集成深度确认）

**现状**：知识注入流程未集成图谱扩展，仅返回向量检索结果。

**修复目标**：向量检索命中后，沿图谱边关系扩展关联 entries 一并注入。

**验收标准：**

- AC1：`kivo-intent-injection` handler 的知识注入流程在向量检索返回 top N 结果后，对每个命中 entry 查询 graph_edges，找到 weight 超过配置阈值的强关联 entries。
- AC2：图谱扩展结果与向量检索结果去重后合并。数量由权重阈值动态决定，设置防爆上限（默认 20 条）防止撑爆 context。禁止硬编码固定返回数量。
- AC3：图谱扩展的 entries 按 edge weight 降序排列，优先注入关联强度最高的条目。
- AC4：注入结果中标注每条 entry 的来源路径（`vector_match` 或 `graph_expansion`），写入注入日志便于调试。
- AC5：weight 阈值可通过 KIVO 配置调整，默认值基于当前图谱 edge weight 分布的 P75 值。
- AC6：当图谱数据不可用（graph_edges 为空或查询失败）时，降级为纯向量检索结果，不阻断注入流程。错误写入日志但不抛异常。

### FIX-06：FR-N06 批量萃取质量治理（集成到 cron 流程）

**现状**：批量提取流程未集成质量门禁，提取结果直接入库。

**修复目标**：cron 批量提取（`kivo extract-sessions`）的产出经过质量门禁过滤后再入库。

**验收标准：**

- AC1：`kivo extract-sessions` 命令的产出在写入 entries 表前，逐条经过统一质量门禁（FR-N05）过滤。门禁不通过的条目不入库。
- AC2：质量门禁默认开启。`--no-quality-gate` 参数可关闭门禁（仅调试/迁移），关闭时输出警告日志。
- AC3：萃取完成后输出门禁统计日志：萃取总条目数、门禁通过数、拒绝数、合并数，以及各拒绝原因的分布（如：重复、不满足三重测试、粒度过粗等）。
- AC4：门禁拒绝率超过可配置阈值（默认 80%）时，输出告警提示萃取源质量异常，建议检查 session 筛选条件。
- AC5：门禁集成点在 `scripts/kivo-session-extract-cron.sh` 调用链中，确保 cron 自动执行时门禁生效，不依赖手动传参。

### FIX-07：FR-N09 碎片聚合治理（完整实现 LLM 驱动合并）

**现状**：碎片聚合逻辑未完整实现。

**修复目标**：定期扫描碎片条目，执行 LLM 驱动的聚合→抽象→规律提炼→有效性评估。

**验收标准：**

- AC1：碎片识别：用 BGE 向量检索找到语义相近的条目簇（cosine similarity 超过动态阈值的候选对），再用 LLM 精判这些候选是否属于同主题碎片。禁止关键词匹配/正则/FTS5/规则引擎承担语义理解职责。
- AC2：碎片聚合：对识别出的碎片簇调用 LLM 执行语义聚合——多条碎片→一条抽象知识。聚合目标是提炼跨场景、跨时间的规律和原则，不是简单拼接。
- AC3：聚合产出的知识条目必须通过 FR-A07 三重测试（时效性、跨场景、抽象性）。不通过的聚合结果不入库，原始碎片保留。
- AC4：聚合完成后删除原始碎片，删除操作记录聚合来源（哪些碎片 ID 被聚合为哪条新知识 ID）。
- AC5：有效性评估由 LLM 执行，评估维度：当前环境下是否成立、是否被更新理解 supersede、指导的行为是否仍正确。
- AC6：CLI 命令 `kivo governance aggregate` 可手动触发，支持 `--domain` 限定范围、`--dry-run` 预览。cron 自动调度频率可配置（默认每天一次）。
- AC7：执行完成后输出报告：扫描总条目数、识别碎片簇数、聚合产出新知识数、删除碎片数、聚合失败数。
- AC8：禁止硬编码定量（如"每次最多聚合 N 条""相似度阈值 0.X"），聚合触发和范围由 LLM 语义判断决定。

### FIX-08：FR-W04 活动流断线重连（服务端事件缓存）

**现状**：活动流支持实时推送，但客户端断线后无法补发断线期间的事件。

**修复目标**：服务端缓存事件，客户端断线重连后自动补发。

**验收标准：**

- AC1：服务端维护一个有界事件缓存（环形缓冲区或等效结构），缓存最近 N 条事件（N 可配置，默认 1000）。缓存存储在内存中，服务重启后清空（不持久化）。
- AC2：每条事件分配单调递增的序列号（`eventId`）。客户端连接时携带 `lastEventId` 参数。
- AC3：服务端收到带 `lastEventId` 的连接请求后，从缓存中找到该 ID 之后的所有事件，按序补发给客户端，补发完成后切换为实时推送。
- AC4：若 `lastEventId` 已不在缓存中（断线时间过长），服务端返回特殊信号告知客户端"历史已丢失"，客户端执行全量刷新。
- AC5：使用 Server-Sent Events（SSE）协议，`id` 字段携带 eventId，客户端浏览器原生支持自动重连和 `Last-Event-ID` 头。
- AC6：前端 EventSource 连接断开后自动重连（浏览器原生行为），重连时自动携带 `Last-Event-ID`，无需额外代码。

### FIX-09：FR-W08 文档导入原文溯源（前端交互）

**现状**：文档导入提取结果缺少来源段落标注和原文查看交互。

**修复目标**：每条提取结果标注来源段落，用户可点击查看原文上下文。

**验收标准：**

- AC1：文档导入提取结果列表中，每条结果显示来源段落定位标签（如"第 3 段"或"第 12-15 行"），标签可点击。
- AC2：点击来源标签后展开/弹出原文上下文面板，高亮显示该条知识提取自的原文段落，上下各展示 1-2 个相邻段落作为上下文。
- AC3：后端提取 API 返回的每条结果包含 `sourceRange` 字段（段落起止位置或行号范围），前端据此定位原文。
- AC4：原文上下文面板支持关闭，关闭后回到提取结果列表视图。
- AC5：大文档（超过 100 段）的原文面板采用虚拟滚动或按需加载，不一次性渲染全文。
- AC6：原文面板中高亮的段落使用视觉区分（背景色标记），与普通上下文段落区分明显。

### FIX-10：FR-K01 管线阶段编排（统一编排入口）

**现状**：管线阶段分散在各模块中，缺少统一编排入口。

**修复目标**：提供统一的管线编排模块，所有提取管线通过该入口注册和执行。

**验收标准：**

- AC1：新增统一管线编排模块（如 `pipeline-orchestrator.ts`），定义管线阶段接口：每个阶段实现 `execute(input) → output` 和 `name` 属性。
- AC2：对话提取管线通过编排器注册五步阶段：原子分解+去上下文化 → 准入门禁 → cosine 去重 → 素材暂存 → 抽象聚合。文档提取管线注册完整阶段（提取 → 分析中间产物 → 分类 → 冲突检测 → 合并检测 → 入库）。
- AC3：阶段之间通过事件驱动流转，编排器负责将上一阶段的 output 传递给下一阶段的 input。
- AC4：任一阶段失败时，编排器中断当前管线实例并记录失败上下文（阶段名、输入数据摘要、错误信息），不影响其他独立管线实例。
- AC5：管线支持扩展——新增处理阶段只需调用编排器的注册方法（如 `pipeline.register(stage, position)`），不修改已有阶段逻辑。
- AC6：编排器提供管线执行状态查询接口：当前活跃管线数、各阶段处理计数、失败计数。
- AC7：现有分散在各模块中的管线调用逻辑迁移到通过编排器执行，消除直接调用各阶段的散落代码。

## 实施建议

- FIX-01 和 FIX-02 为 P0，应优先实施。FIX-01 修改范围集中在单个 handler 文件，风险可控。FIX-02 涉及前端组件重写，工作量较大。
- FIX-10（管线编排）虽为 P1，但作为基础设施建议早期实施，后续 FIX-01/FIX-06 可直接基于编排器集成。
- FIX-04 和 FIX-05 有依赖关系，应按序实施。
