## 1. 产品愿景与定位

KIVO(Knowledge Iteration & Vibe Orchestration)= Agent 知识平台 + 个人知识工作台。

产品定位:用户的认知进化伙伴。系统定位:Agent 知识平台。

KIVO 的用户交互体验目标是远超市面上现有的知识库和 AI 笔记应用(Obsidian、Notion、Mem、Reflect 等)。前端交互体验是 KIVO 作为平台的核心竞争力,不是附属品。具体体现在:交互式知识图谱探索(力导向布局、节点着色、缩放聚焦)、调研管理驾驶舱(缺口报告、任务队列、进度追踪)、文档导入流水线(分段提取、逐条确认、原文溯源)。全站采用浅色主题(白色背景 + 黑色文字)。

KIVO 覆盖知识的完整生命周期--从多源提取、结构化存储、语义检索、冲突解决、自主调研到图谱洞察。Agent 通过 API 消费知识,用户通过 Web 工作台录入、探索、研究和治理知识资产。工作台是独立产品对象,围绕四个核心入口构建:意图知识库、领域知识库(LLM Wiki)、知识图谱、知识搜索。

KIVO 解决的核心问题:

- Agent 的知识散落在对话、记忆文件、配置和临时笔记里,没有结构、没有版本、没有主动补盲机制。
- 用户的知识散落在文档、笔记、浏览器书签和脑子里,缺乏结构化沉淀、关联发现和缺口洞察。
- 知识的录入、分析、沉淀、探索、调研形成的完整闭环,当前没有一个系统能端到端覆盖。

设计硬约束:

- 知识先结构化再存储,不存原始文本堆。
- 知识冲突必须自动解决,不允许新旧矛盾共存。治理 cron 负责检测和解决冲突,不暴露为用户手动操作。
- 调研能力自主触发,不依赖用户手动投喂。
- 规则分发按需订阅,不全量灌注。
- 代码不硬编码路径、凭据和环境细节,保持工程质量;但产品层面 KIVO 只服务于 OpenClaw 宿主环境,不承诺支持其他平台。
- 引擎层和工作台层独立演进,工作台是独立产品对象,不是引擎的附属页面。

### 成功指标

- 知识检索命中率 ≥ 85%(Agent 查询时返回相关知识条目的比例)。度量方法:基于人工标注测试集,检索结果 Top-K 中包含标注相关条目的查询占比。
- 缺口检测覆盖率 ≥ 70%(已知盲区中被自动发现的比例)。
- 知识冲突自动解决率 100%（冲突发生后由治理 cron 自动解决,不允许悬而未决）。
- 规则分发到达率 ≥ 95%(订阅者收到规则更新的比例)。
- 首次知识旅程完成时间 ≤ 10 分钟(空库用户从安装到看到系统价值)。

## 2. 目标用户画像

### 2.1 个人知识工作者

拥有大量文档、笔记、研究资料,需要一个系统帮助整理、关联和发现知识盲区。关注知识的录入体验、图谱探索和深度研究闭环。典型场景:导入一批论文,系统自动提取关键概念并建立关联,发现知识缺口后自动调研补齐。

### 2.2 Solo Founder / 独立产品操盘者

用 Agent 推进产品研发、运营和决策。需要 Agent 记住历史决策、方法论和经验教训,不在同一个坑上反复踩。需要 Agent 能自主调研行业动态和技术趋势,不依赖手动投喂。

### 2.3 Agent 开发者

构建和维护 Agent 系统。需要统一的知识管理接口,让不同 Agent 共享知识而不互相污染。需要规则分发机制,确保系统规则变更能及时传达到所有相关 Agent。

### 2.4 知识治理视角

同一用户在治理知识资产时的关注点:知识库有多少盲区、多少过时条目待清理、检索命中率趋势如何。需要控制哪些 Agent 能访问哪些知识域。

## 3. 核心概念

### 3.1 核心对象类型

- **Knowledge Entry**:知识条目,KIVO 管理的最小知识单元。包含类型、内容、来源、版本、状态和关联关系。粒度标准:一条 Knowledge Entry = 一个可独立验证的知识断言。
- **Knowledge Type**:知识类型枚举--fact、methodology、decision、experience、intent、meta。详见 §3.3。
- **Source Reference**:来源引用,记录知识的出处(对话 ID、文档路径、URL、规则文件路径等)。
- **Conflict Record**:冲突记录,记录两条知识之间的冲突、解决策略和解决结论。
- **Research Task**:调研任务,由缺口检测生成,包含目标、范围、策略、预算和状态。
- **Gap Report**:缺口报告,记录知识库的盲区分析结果和对应的调研建议。
- **Rule Entry**:规则条目,可分发的系统规则。管"Agent 必须遵守什么约束",与 Knowledge Entry(管"Agent 知道什么")职责分离。
- **Subscription**:订阅关系,记录 Agent 与规则集之间的绑定。
- **Distribution Record**:分发记录,记录规则推送的目标、时间和确认状态。
- **Research Queue**:调研队列,承载待执行和执行中调研任务的调度容器。有 active 和 silent 两种模式。
- **Analysis Artifact**:分析中间产物,知识提取过程中产生的结构化分析结果,包含候选实体、概念、关联建议、冲突建议和调研建议。
- **Knowledge Graph**:知识图谱,基于 Knowledge Entry 之间的关联关系构建的网络结构,支撑洞察发现和可视化探索。
- **Domain Purpose**:知识域目标声明,定义每个知识域的目标、关键问题、非目标和研究边界,指导知识采集和治理方向。

### 3.2 知识条目状态

知识条目只有一种状态:**active**(生效,可被检索和引用)。

不存在 pending、archived、superseded、deprecated 等中间状态。提取质量不够的知识直接丢弃,不入库;过时的知识直接删除,不保留历史。知识库只保留当前有效的知识。

设计决策(2026-05-06 用户确认):多状态生命周期增加了无意义的复杂度。没有人会去逐条审核 pending 条目,archived 条目占空间但永远不会被查看。简化为单状态,提取时直接决定入库或丢弃。

### 3.3 知识类型判别规则

| 类型 | 定义 | 判别关键 | 正例 | 反例 |
|------|------|----------|------|------|
| fact | 对客观世界的陈述,可独立验证真假 | 描述"是什么",不含主观判断 | "Node.js 22 默认启用 ESM" | "应该用 Node.js 22"(decision) |
| methodology | 可复用的做事方法,已抽象为通用步骤 | 不绑定某次具体实践 | "先写失败测试,再写最小实现" | "上次我先写了测试效果不错"(experience) |
| decision | 在多个方案中做出的明确选择 | 含"选择 X 而不选 Y"结构 | "本项目采用 PostgreSQL 而非 MongoDB" | "PostgreSQL 支持事务"(fact) |
| experience | 某次具体实践的经历和教训 | 绑定特定上下文,未抽象为通用方法 | "上次迁移 ESM 时因没检查依赖兼容性回滚了两次" | "迁移 ESM 前应先检查依赖兼容性"(methodology) |
| intent | 用户的偏好、目标、约束或期望 | 表达主观意愿,可能随时间变化 | "用户要求代码注释用中文" | "代码注释必须用中文"(若为强制规范则进 Rule Entry) |
| meta | 关于知识管理本身的反思 | 主题是知识过程本身 | "连续三次纠偏同一类问题时应主动生成调研任务" | "每周回顾一次项目进度"(methodology) |

## 4. 功能需求(FR)

### 域 A:知识提取(Knowledge Extraction)

负责从各类来源中提取结构化知识。

#### FR-A01:对话知识提取

从 Agent 与用户的对话中自动提取结构化知识。提取管线精简为五步:原子分解+去上下文化 → 行为变化测试+三重测试(准入门禁) → 向量去重 → 素材暂存 → 抽象聚合入库。

- AC1:系统能从对话中识别并提取六类知识:fact、methodology、decision、experience、intent、meta。
- AC2:提取结果包含 LLM 归纳的短标题(≤20字,语义完整的短句或短语,是对知识点的提炼概括,禁止硬截断)、LLM 归纳的结构化内容描述(不限字数,要求清晰讲透知识点的含义、适用场景和边界,确保被检索命中后能为 Agent 提供足够上下文做出正确判断)、知识类型、置信度、来源引用、提取时间和 2-3 个相似表述/泛化例句。标题和内容必须经过 LLM 抽象归纳,不能直接照搬原文。置信度仅用于决定是否入库,入库后不保留为条目属性。禁止 description 复制 title;description 必须比 title 提供更多信息。
- AC3:置信度低于可配置阈值的提取结果直接丢弃,不入库。
- AC4:提取完成后,置信度达标的条目自动写入知识库;低置信度条目直接丢弃。
- AC5:提取过程产出 Analysis Artifact(域 L),记录分析中间结果,支持审计和回放。
- AC6:原子分解必须同时完成去上下文化(decontextualization)--解析代词指代、补全省略的实体名称、将隐含的上下文信息显式化。产出的每条原子知识必须脱离原始对话上下文后仍可独立理解。原子分解与去上下文化在同一个 LLM 调用中一步完成,不拆为两个阶段。
- AC7:行为变化测试是知识入库的唯一准入门禁。判断标准由三重测试组成,全部通过才允许入库,任一不通过直接丢弃:
  - 时效性测试:「一个月后这条信息还有效吗?」临时状态、一次性决策、排查中间步骤等时效性信息不通过。
  - 跨场景测试:「换一个完全不同的项目/场景,它还适用吗?」绑定特定项目、特定时间窗口的信息不通过。
  - 抽象性测试:「去掉具体的人名、项目名、时间,它是否仍有指导价值?」未经抽象的具体事件记录不通过。
  - 行为影响测试:「如果这条知识不存在,agent 会做出不同的(错误的)决策吗?」回答为「否」不通过。
  三重测试由 LLM 语义判断执行,禁止使用关键词匹配、正则表达式、FTS5、规则引擎承担语义理解职责。
  明确的负例(以下内容禁止入库):一次性调度决策(「先做A再做B」)、临时优先级排序(「这周先搞官网」)、排查中间步骤(「加日志定位根因」)、任务派发指令(「codex 去做这个」)、未经抽象的具体事件记录(「今天 cc 超时了」)、配置描述(「当前用 penguin provider」)、待办事项(「需要加个监控脚本」)。
  明确的正例(这些是知识):跨场景可复用的原则(「关键词匹配不能承担语义理解职责」)、长效有效的方法论(「修复问题先堵根因再修症状」)、抽象的理解模型(「LLM 始终可用,不存在无 LLM 降级场景」)。
- AC8:应该提取的知识类型--用户私有决策(LLM 不可能自己知道)、用户私有术语/黑话、反复出现的 badcase(LLM 知道但做不到)、用户特有的偏好和约束、容易被 context 稀释遗忘的关键约束。
- AC9:禁止提取的内容--通用常识(LLM 本来就会)、概念解释(LLM training data 里有)、通用方法论教程(LLM 本身就知道怎么做)、非私有的事实性描述。此类内容即使出现在对话中也必须跳过,不生成知识条目。
- AC10:提取粒度为原子级--单次提取产出的是原子级知识片段:一条决策、一条纠偏、一条事实、一条偏好。每条原子知识通常一两句话能说清。禁止从对话中一次性摘出大段文字当知识条目。禁止产出百科式解释、教程式方法论、步骤式操作指南。
- AC11:复杂知识只能通过合并加工产出,不能在提取阶段直接生成。流程:多次提取原子片段 → 发现多条指向同一主题 → 合并加工成一条更完整的知识(由域 N 治理流程或域 C 迭代流程负责)。合并后的知识条目可以较长,但必须是加工产物,不是原始摘录。
- AC12:入库前向量去重--对候选知识做 embedding,与已有知识库计算 cosine 相似度,>= 0.92 视为重复,重复条目跳过入库。去重阈值 0.92 可通过 kivo.config.json 的 `extraction.dedup.threshold` 配置。

#### FR-A02:文档知识提取

从文档、论文、书籍等文本材料中提取结构化知识。

- AC1:支持 Markdown、PDF、纯文本、EPUB、网页正文等常见文本格式。
- AC2:提取结果保留原文引用位置,支持溯源。
- AC3:长文档支持分段提取,每段独立标注知识类型和关联关系。
- AC4:提取完成后,置信度达标的条目自动写入知识库;低置信度条目进入待确认队列。
- AC5:提取过程产出 Analysis Artifact(域 L),记录候选实体、概念、关联建议等中间结果。

#### FR-A03:规则提取

从系统配置、AGENTS.md、SOUL.md 等治理文件中提取可分发的规则条目。

- AC1:规则条目包含规则内容、适用范围、优先级和生效条件。
- AC2:规则变更时自动检测差异,生成变更事件。
- AC3:规则之间的冲突和覆盖关系被显式记录。

#### FR-A04:个人知识录入

支持用户通过多种入口主动录入知识。

- AC1:支持手动录入--用户在 Workbench 中直接创建知识条目,填写标题、内容、类型和标签。
- AC2:支持文件导入--用户上传文档文件,系统自动提取知识(复用 FR-A02 文档提取管线)。
- AC3:支持 URL 抓取--用户输入网页 URL,系统抓取正文并提取知识。
- AC4:支持对话沉淀--用户在 Workbench 中标记对话片段为"值得保留",系统提取并入库。
- AC5:支持批量文件夹导入--用户指定本地目录,系统递归扫描并逐文件提取知识,展示导入进度。
- AC6:所有录入入口产出的知识条目走同一条管线(提取 → 冲突检测 → 入库),不绕过一致性保障。

#### FR-A05:会话历史知识萃取(Session Transcript Knowledge Extraction)

从用户与 Agent 的历史对话归档中批量提取有价值的知识。采用两阶段架构:

**Stage 1:素材采集** - 从对话中识别有潜在知识价值的片段,产出「素材」暂存在临时区域,不直接入库。素材是原始的、未经抽象的候选片段。

**Stage 2:抽象聚合** - 对同主题的多条素材进行综合,产出经过抽象的知识条目。只有 Stage 2 的产出才能进入入库流程(经三重测试 FR-A07)。聚合维度:同一主题在不同时间/场景下的多次表达 → 收敛为一条跨场景原则。已有知识库中存在相关条目时,优先更新/合并而非新增。

FR-A01 处理实时对话流中的知识提取;FR-A05 处理已归档的 session 文件,覆盖产品思路、决策推理、纠偏意图等高价值但未被沉淀的对话内容。两者互补:FR-A01 管增量,FR-A05 管存量。

与 FR-E04(意图知识提取)的关系:FR-E04 专注从对话中提取 intent 类型知识(正例/负例/信号);FR-A05 提取所有知识类型(fact/methodology/decision/experience/intent/meta),覆盖面更广,提取粒度更粗。两者可共存,FR-E04 的意图提取结果可作为 FR-A05 的去重参考。

依赖 FR-B05(多维知识标签体系):提取的知识条目使用三维标签(nature/function/domain)而非单一 type 字段。

前置条件:用户已配置可用的 embedding provider(如 Ollama + bge-m3、OpenAI embedding 等)。未配置 embedding provider 时,命令报错并输出配置引导,不静默跳过。

- AC1:CLI 入口 `kivo extract-sessions` 可用,`--help` 输出命令说明、参数列表和使用示例。
- AC2:对真实 OpenClaw session JSONL 文件运行,能从用户消息中提取结构化素材并暂存到临时区域(staging),经 Stage 2 聚合后才写入知识库。
- AC3:噪音过滤有效--纯操作性对话(重启服务、查状态、部署、文件操作等)不被提取为知识条目。噪音过滤采用关键词规则,不依赖 LLM。
- AC4:未配置 embedding provider 时,命令报错退出(非零退出码),输出明确的配置引导信息(推荐 Ollama + bge-m3 方案及替代方案),引导用户完成配置后重试。
- AC5:`--dry-run` 模式只统计可提取的知识条目数量和分类分布,不写入数据库。
- AC6:增量处理--系统记录已处理的 session 文件标识,重复运行不重复提取。支持 `--since DATE` 参数限定处理范围。
- AC7:提取的知识条目携带三维标签(nature/function/domain),标签由 LLM 在提取时同步生成。
- AC8:提取流程为:JSONL 解析 → 用户消息提取 → 关键词噪音过滤 → BGE embedding 生成 → 语义聚类去重 → LLM 结构化知识提取(含三维标签) → 标准知识管线入库。
- AC9:`--limit N` 参数限制单次处理的 session 文件数量,用于控制 LLM 调用成本。
- AC10:定时萃取的用户活跃判定必须基于外部渠道的真实用户消息(如 Gateway 日志中的飞书 DM 入站记录),禁止使用 session 文件 mtime 作为判定依据。session 文件 mtime 会被系统内部事件(子 Agent completion event、heartbeat、内部消息路由)更新,与用户是否在对话无因果关系。
- AC11:「有对话但未提取」通知(written=0 告警)仅在同时满足以下两个条件时发送:1指定时间窗口内存在来自外部渠道的真实用户消息;2本轮萃取 written=0。系统内部事件不计入用户活跃判定,不触发此通知。
- AC12:禁止使用消息字数、字符数、段落长度、消息条数作为是否提取的判断条件。禁止在代码中设置 MIN_SEGMENT_LEN、MIN_MESSAGE_LENGTH、MIN_QUEUE_SIZE 或任何基于数量阈值的预过滤逻辑。知识价值与消息长度无因果关系--用户最高价值的纠偏可能只有几个字。正确做法:将用户消息全量聚合后交给 LLM 语义判断是否有提取价值,由 LLM 决定「无高价值内容」而非代码预过滤。唯一允许的预过滤:纯系统生成内容(heartbeat、system prompt injection、内部路由消息等非用户产出内容)。
- AC13:提取流程中的去重判断必须使用 BGE 向量余弦相似度,禁止使用 FTS5 token 匹配、关键词重叠检测或正则表达式做去重。extract-queue-worker 的去重逻辑适用此约束。
- AC14:萃取产出必须遵守 FR-A01 AC6-AC11 定义的准入标准、原子粒度和三重校验。LLM 提取 prompt 中必须包含准入判定规则和原子粒度约束,从源头控制产出质量,而非仅依赖下游门禁过滤。
- AC15:Stage 1 产出的素材不得直接写入 entries 表。素材暂存在独立的 staging 区域(staging_materials 表或等效存储),等待 Stage 2 聚合处理。
- AC16:Stage 2 抽象聚合必须由 LLM 执行语义综合,禁止使用关键词匹配、正则表达式、FTS5、规则引擎承担语义理解职责。聚合逻辑:对同主题的多条素材进行语义综合,提炼出跨场景、跨时间的抽象原则。聚合的目标是产出更少、更精、更抽象的知识条目,而非更多。
- AC17:聚合后的知识条目必须通过 FR-A07 定义的三重测试(时效性、跨场景、抽象性),全部通过后才能进入入库流程。
- AC18:单次提取的素材数量不设硬编码上限,由 LLM 判断「是否有提取价值」。禁止在代码中设置 MAX_MATERIALS_PER_SESSION 或任何基于数量的截断逻辑。
- AC19:聚合时已有知识库中存在相关条目时,优先更新/合并已有条目而非新增。合并策略:保留更抽象、更完整的表述,来源引用取并集。
- AC20:Stage 2 聚合的触发时机:每次 Stage 1 完成后自动触发一次聚合,对 staging 区域中所有未聚合的素材执行语义聚类和抽象提炼。聚合完成后清除已消费的素材。

#### FR-A06:实时知识提取(Real-time Knowledge Capture)

从 Agent 与用户的实时对话中自动捕获高价值知识,每轮对话结束后自动分析并提取。FR-A01 定义对话知识提取的通用能力;FR-A06 定义自动化触发机制和高价值知识的识别策略,确保纠偏、决策、新规则、事实更新、经验教训等关键知识不因对话结束而丢失。

- AC1:每轮对话结束后自动触发知识提取,分析当前轮次的用户消息和 Agent 回复。
- AC2:LLM 识别并提取高价值知识类型:纠偏(correction)、决策(decision)、新规则(rule)、事实更新(fact_update)、经验教训(lesson_learned)。
- AC3:提取的知识条目经统一质量门禁(FR-N05)过滤后再入库,不绕过门禁直接写入。
- AC4:知识识别必须使用 LLM 语义理解,禁止关键词匹配或规则引擎。
- AC5:提取结果包含来源追溯信息:session_id、message_id、对话时间戳,支持从知识条目反查原始对话上下文。
- AC6:hook 执行异步化,不阻塞对话主流程,提取失败不影响用户正常对话。
- AC7:同一轮对话中提取的多条知识条目各自独立经过门禁,不因其中一条被拒绝而影响其他条目。
- AC8:禁止使用消息字数、字符数、消息条数作为是否触发提取或是否入队的判断条件。禁止在代码中设置 MIN_MESSAGE_LENGTH、MIN_QUEUE_SIZE 或任何基于数量阈值的预过滤逻辑。每条用户消息到达时立即入队并触发提取流程,由 LLM 语义判断是否包含高价值知识,而非由代码根据长度或条数预先过滤。唯一允许的预过滤:纯系统生成内容(heartbeat、system prompt injection、内部路由消息等非用户产出内容)。
- AC9:实时提取必须遵守 FR-A01 AC6-AC11 定义的准入标准、原子粒度和三重校验。提取 prompt 中必须包含准入判定规则和原子粒度约束,从源头控制产出质量,而非仅依赖下游门禁过滤。

#### FR-A07:知识定义与准入边界(Knowledge Definition & Admission Boundary)

定义「知识」的本质和边界,为所有入库路径提供统一的准入判定标准。

知识的定义:经过抽象、聚合、萃取后形成的长效理解模型。跨时间、跨场景可复用。缺了它 agent 会系统性犯错。知识库存在的唯一目的是让 agent 在未来做出更好的判断。每条知识必须能回答:「它让 agent 在什么场景下避免了什么错误」。回答不了的就不该存在。

知识的判定标准(三重测试,全部通过才入库):
1. 时效性测试:一个月后这条信息还有效吗?
2. 跨场景测试:换一个完全不同的项目/场景,它还适用吗?
3. 抽象性测试:去掉时间和场景上下文后,它是否仍有指导价值?

明确的负例(以下不是知识,禁止入库):
- 一次性调度决策(「先做A再做B」)
- 临时优先级排序(「这周先搞官网」)
- 排查中间步骤(「加日志定位根因」)
- 任务派发指令(「codex 去做这个」)
- 未经抽象的具体事件记录(「今天 cc 超时了」)
- 配置描述(「当前用 penguin provider」)
- 待办事项(「需要加个监控脚本」)

明确的正例(这些是知识):
- 跨场景可复用的原则(「关键词匹配不能承担语义理解职责」)
- 长效有效的方法论(「修复问题先堵根因再修症状」)
- 抽象的理解模型(「LLM 始终可用,不存在无 LLM 降级场景」)

- AC1:所有入库路径必须执行三重测试(时效性、跨场景、抽象性),任一不通过则拒绝入库。三重测试与 FR-A01 AC7 的行为变化测试共同构成完整的准入门禁体系。
- AC2:三重测试由 LLM 语义判断执行,禁止使用关键词匹配、正则表达式、FTS5、规则引擎承担语义理解职责。所有语义判断必须走 LLM 或向量检索。
- AC3:负例清单可扩展,通过配置文件或知识库自身维护。新发现的负例模式可动态追加到负例库中。
- AC4:准入判定的 LLM prompt 必须包含完整正例和负例,确保 LLM 有充分的边界参考做出判断。
- AC5:每条知识入库时必须能回答「它让 agent 在什么场景下避免了什么错误」。回答不了的条目拒绝入库。这是知识库存在的目标结果导向约束。

### 域 B:知识存储与检索(Knowledge Store & Retrieval)

负责知识的结构化存储、版本管理和高效检索。

#### FR-B01:结构化知识存储

按知识类型分类存储,支持版本追踪。

- AC1:每条知识条目至少包含:ID、类型、内容、来源、创建时间、更新时间、版本号、状态、所属知识域。
- AC2:知识条目只有 active 状态。
- AC3:同一知识条目的历史版本可追溯。
- AC4:KnowledgeEntry 支持 domain-specific metadata 扩展,不同域可在 metadata 字段中挂载结构化子 schema。

#### FR-B02:语义检索

支持基于语义相似度的知识检索。

- AC1:Agent 查询时,系统返回语义最相关的知识条目,按相关度排序。
- AC2:检索结果包含知识条目内容、类型、来源和相关度评分。
- AC3:支持按知识类型、时间范围、来源、知识域等条件过滤。
- AC4:检索结果受 Domain Purpose(域 M)约束--当知识域配置了目标声明时,检索排序考虑条目与域目标的相关度。
- AC5:语义检索的存储层实现(sqlite-provider 的 search/findSimilar 等方法)必须走向量检索路径,禁止使用 FTS5 MATCH 做语义检索。FTS5 仅允许用于按 ID 精确查找等非语义场景。
- AC6:CLI 查询命令(kivo query / kivo search)必须走向量语义检索,禁止使用 FTS5 MATCH 或关键词匹配作为检索实现。
- AC7:检索结果的相关度评分必须基于向量余弦相似度,禁止使用关键词命中率或 BM25 评分。

#### FR-B03:知识关联

维护知识条目之间的关联关系。

- AC1:支持的关联类型至少包括:补充(supplements)、替代(supersedes)、冲突(conflicts)、依赖(depends_on)。
- AC2:新知识入库时自动检测与已有知识的潜在关联。
- AC3:关联关系可被检索时利用,提升返回结果的完整度。
- AC4:关联关系同时作为知识图谱(域 G)的边数据,支撑图谱构建和洞察分析。

#### FR-B04:语义向量生成与缓存

知识条目入库时自动生成语义向量,支撑语义检索和冲突检测。

- AC1:知识条目入库时自动触发 Embedding 生成,生成结果与条目关联存储。
- AC2:Embedding 结果本地缓存,相同内容不重复生成。条目内容变更时自动失效旧缓存并重新生成。
- AC3:Embedding 生成失败时,条目仍正常入库,标记为"待补建索引",Provider 恢复后自动补建。
- AC4:语义检索不设降级路径--禁止实现「embedding 不可用时降级为关键词/FTS5/元数据过滤」的逻辑。若 embedding provider 不可达或生成异常,应报错并输出修复引导(检查 provider 连通性、模型是否加载),不降级为非语义检索。
- AC5:向量索引支持持久化存储(与知识库同一存储引擎),进程重启后自动加载已有索引,不丢失已建向量数据。
- AC6:向量索引采用内存加速 + 持久化兜底策略--搜索在内存中执行,写入时同步持久化,启动时从持久化存储加载到内存。

#### FR-B05:多维知识标签体系(Multi-dimensional Knowledge Tagging)

将知识分类从单一 type 字段升级为三个正交维度,支持多角度检索和组织。

与 FR-B01(结构化知识存储)的关系:FR-B05 扩展 FR-B01 的 schema,新增三个字段,不修改已有字段。FR-B01 的 AC1 中「类型」字段保持不变,FR-B05 在其基础上叠加更细粒度的分类维度。

与 FR-K02(知识分类与路由)的关系:FR-K02 的分类器输出从单一 type 扩展为 type + nature + function + domain 四个字段。FR-K02 的路由逻辑可基于任意维度组合做差异化处理。

三个标签维度:
- **nature**(知识本质):fact / concept / rule / procedure / heuristic。描述知识「是什么形态」。
- **function**(知识功能):routing / quality_gate / context_enrichment / decision_support / correction。描述知识「用来做什么」。
- **domain**(知识领域):开放标签,LLM 自动归类或用户手动指定。描述知识「属于哪个领域」。

向后兼容策略:保留原有 type 字段(fact/methodology/decision/experience/intent/meta),新写入的知识同时填充 type 和三维标签。已有知识的 type 字段不受影响,三维标签字段为空直到执行 retag。

- AC1:DB schema 新增 nature、function、domain 三个字段,字段类型为字符串,允许为空(向后兼容)。
- AC2:新写入的知识条目(通过任何入库路径)自动由 LLM 填充三维标签,标签生成集成在知识管线的分类阶段(FR-K02)。
- AC3:CLI 命令 `kivo retag` 可用,对已有知识条目批量补打三维标签。`--dry-run` 模式预览标签变更(输出每条知识的当前 type 和建议的 nature/function/domain),不写入数据库。
- AC4:已有知识条目的 type 字段在 retag 过程中不被修改,向后兼容。
- AC5:`kivo search` 支持按维度组合过滤:`--nature <value>`、`--function <value>`、`--domain <value>`,多个维度参数为 AND 关系。
- AC6:Web 工作台知识列表(FR-W02)支持按 nature、function、domain 三个维度独立筛选,筛选控件与现有的 type/status 筛选并列。
- AC7:`kivo retag` 支持 `--limit N` 参数限制单次处理条目数,支持 `--domain <value>` 参数限定 retag 范围。
- AC8:retag 完成后输出统计报告:处理条目数、各维度标签分布、失败条目数及原因。

### 域 C:知识迭代(Knowledge Iteration)

负责知识的更新、冲突解决和一致性维护。

#### FR-C01:冲突检测与解决

新知识与旧知识冲突时自动检测并解决。

- AC1:系统能检测语义冲突和规则冲突。语义冲突的信号类型包括:同一实体的属性值矛盾、同一决策的结论相反、同一事实的时间版本冲突。规则冲突指同一场景下的矛盾规则。
- AC2:冲突检测采用两阶段策略--粗筛阶段基于 Embedding 相似度识别候选冲突对(阈值可配置),精判阶段由 LLM 对候选对做语义矛盾判定。
- AC3:冲突解决策略包括:时间优先(新知识替代旧知识)、来源优先(高权威来源优先)。所有冲突由治理 cron 自动解决,不暴露为用户手动操作。
- AC4:冲突解决过程和结论被记录,支持复盘。

#### FR-C02:知识过期与清理

自动识别过时知识并处理。

- AC1:系统能基于时间衰减、引用频率和外部验证标记过时知识。
- AC2:过时知识直接物理删除,不保留历史。删除前记录快照用于审计追溯。
- AC3:清理操作生成报告,记录被清理的条目和清理原因。

#### FR-C03:知识合并

来自不同来源的同主题知识自动合并。

- AC1:系统能识别不同来源对同一主题的描述,合并为一条综合知识条目。触发条件:同一主题下存在 2 条及以上无冲突、语义互补的知识条目,且 Embedding 相似度超过可配置阈值。
- AC2:合并策略按知识类型区分--fact 和 experience 类型取并集合并;decision 类型禁止自动合并,必须人工确认;methodology 类型合并后需人工确认。
- AC3:合并后的条目保留所有来源引用。
- AC4:合并操作可回退。

### 域 D:自主调研(Autonomous Research)

负责主动发现知识缺口并发起调研填补。

#### FR-D01:缺口检测

自动发现当前知识库的盲区和缺口。

- AC1:系统能基于 Agent 查询未命中记录识别高频盲区。
- AC2:系统能基于知识条目关联关系的结构缺失识别逻辑缺口。
- AC3:系统能基于知识图谱结构信号识别缺口--孤立节点(已入库但几乎不参与任何关联)、稀疏社区(条目不少但互相支撑关系弱)、桥接节点缺失(两个主题群之间缺少连接知识)。
- AC4:缺口检测结果按优先级排序,输出为结构化的调研任务建议。
- AC5:缺口报告中的盲区条目支持一键创建调研任务,与调研队列直接联动。
- AC6:覆盖度分析提供目标基准线,用户可判断当前命中率是否达标。

#### FR-D02:调研任务生成与执行

基于缺口检测结果自动生成并执行调研任务。

- AC1:调研任务包含目标、范围、预期知识类型、搜索策略和完成标准。
- AC2:调研任务定义中包含所需的信息获取方式(web search、文档阅读、论文解析等),由 OpenClaw 宿主环境负责执行。
- AC3:调研结果经过知识提取流程后入库,形成闭环。
- AC4:调研任务有资源预算(时间、API 调用次数),超预算自动终止并报告已获取的部分结果。
- AC5:已完成的调研任务支持"重新调研"操作。
- AC6:调研任务提供详情视图,展示调研过程、中间产出、引用来源等完整上下文。

#### FR-D03:调研优先级与调度

管理调研任务的优先级和执行节奏。

- AC1:调研优先级基于缺口影响面(影响多少 Agent 查询)和紧迫度(是否阻塞当前任务)。
- AC2:调研任务不能抢占用户主动发起的任务资源。
- AC3:调研执行频率可配置,支持静默模式(只记录缺口不执行调研)。

### 域 E:意图理解增强(Intent Enhancement)

负责从对话中自动检测意图信号、提取结构化意图知识,并在任务执行时注入相关知识,提升 Agent 对用户意图的理解。覆盖完整的意图知识生命周期:感知 → 提取 → 存储 → 匹配 → 注入 → 治理。

#### FR-E01:上下文注入

Agent 处理用户请求时,自动注入相关知识作为上下文。

- AC1:注入的知识按相关度排序,总量控制在 token 预算内。token 预算由调用方在检索请求中指定。
- AC2:注入内容包含知识正文和来源标注,Agent 可判断可信度。
- AC3:注入过程对用户透明,不改变用户原始请求。
- AC4:术语注入优先级高于一般知识注入--当预算不足时,优先保障术语注入。
- AC5:知识注入的检索环节必须使用向量语义检索(BGE embedding cosine similarity),禁止使用 FTS、关键词匹配、正则表达式或规则引擎作为注入检索手段。无例外、无 fallback、无降级。
- AC6:知识注入的相关性评分必须基于向量余弦相似度计算,禁止使用关键词命中率、TF-IDF、BM25 或任何基于词频的评分机制。relevance-scorer 组件的评分逻辑只允许使用 BGE embedding cosine similarity。

#### FR-E02:意图消歧

利用历史知识辅助消解用户意图的歧义。

- AC1:当用户输入存在多种理解时,系统基于历史决策和用户偏好知识给出最可能的解释。
- AC2:消歧结果附带置信度和依据的知识条目。
- AC3:消歧失败时,系统生成结构化澄清建议而非猜测。

#### FR-E03:意图信号自动检测

从 Agent 与用户的对话中自动检测意图信号,识别用户的纠偏、强调、声明、规则和偏好表达。

- AC1:系统通过 LLM 语义识别(非关键词匹配)检测五类意图信号:纠偏(correction)、强调(emphasis)、声明(declaration)、规则(rule)、偏好(preference)。
- AC2:每个检测到的信号包含信号类型、置信度评分、原始对话片段引用。
- AC3:置信度低于可配置阈值的信号被丢弃,不进入后续提取流程。
- AC4:信号检测在对话结束后异步执行,不阻塞 Agent 主任务。
- AC5:渐进式披露--L0(安装即用):默认启用所有信号类型,阈值 0.6,用户无需配置;L1(调阈值):用户可调整检测灵敏度、启用/禁用特定信号类型、控制每轮对话最大信号数;L2(自定义信号类型):用户可注册自定义信号类型,提供类型名称、描述和检测 prompt 片段,自定义类型走同一条提取管线。
- AC6:单 Agent 场景兼容--当 OpenClaw 环境只有一个 Agent 时,从主会话对话中收集信号,不依赖子 Agent 事件。

#### FR-E04:意图知识提取与结构化存储

将检测到的意图信号提取为结构化意图知识条目,持久化存储并建立向量索引。

- AC1:信号检测和知识提取合并为一次 LLM 调用,同时输出信号类型和结构化知识,避免多次 LLM roundtrip。
- AC2:提取的意图知识包含:标题、正文内容、正例列表(用户想要的)、负例列表(用户不想要的)、来源对话引用。
- AC2.1:每条意图知识条目附带 5~10 条相似句(paraphrases),即用户可能用来表达同一意图的不同自然语言句子。相似句由 LLM 在提取时同步生成,存储在 similar_sentences 字段(JSON 数组)。相似句用于提升意图匹配的召回率--向量检索时同时匹配 content 和相似句。
- AC2.2:对已有意图条目(similar_sentences 为空),提供 CLI 命令(kivo enrich-intents)批量调用 LLM 补充生成相似句。
- AC2.3:Web 工作台的知识列表和详情页展示意图条目的相似句,列表页用标签样式,详情页用编号列表。
- AC3:提取结果以 Knowledge Entry(type=intent)形式入库,走标准知识管线(冲突检测 → 合并检测 → 入库)。
- AC4:入库的意图知识条目自动生成语义向量并写入持久化向量索引,进程重启后索引自动恢复。
- AC5:支持批量补漏--定时扫描历史对话日志,对未处理的对话片段做批量信号检测和提取。

#### FR-E05:任务派发时意图知识注入

在 Agent 处理用户请求或派发子任务时,自动匹配并注入相关意图知识。

- AC1:Agent 构建 prompt 前,系统用当前消息内容做向量匹配,将相关意图知识注入上下文,注入内容用标记包裹与其他注入隔离。
- AC2:派发子任务时,系统用任务描述做向量匹配,将相关意图知识追加到任务 prompt 中。
- AC3:注入量受 token 预算控制(可配置),预算不足时按相关度排序截断,意图知识优先级高于一般知识。
- AC4:注入内容包含知识正文和来源标注,接收方可判断可信度。
- AC5:注入最低相关度分数可配置,低于阈值的匹配结果不注入,避免噪声。
- AC6:注入路由增加价值判定--向量匹配命中后,系统评估该知识是否填补当前场景下 LLM 的能力盲区(而非仅看语义相似度),只注入 LLM 自身无法覆盖的高价值知识。评估维度:知识的私有性、场景特异性、LLM 在该领域的已知短板。通用常识类知识即使语义匹配度高也不注入(LLM 本来就会的知识只增加 token 消耗,不提升输出质量)。
- AC7:意图知识注入的检索环节必须使用向量语义检索(BGE embedding cosine similarity),禁止使用 FTS、关键词匹配、正则表达式或规则引擎。无例外、无 fallback、无降级。

#### FR-E06:意图知识治理

定时对意图知识库执行聚合分析、去重合并和优先级调整,识别高频主题并清理低效条目。

- AC1:系统定时扫描近 N 天(可配置,默认 7 天)的意图条目,按语义相似度聚类,识别高频主题--被多次独立对话反复强调的规则或纠偏模式。
- AC2:高频主题自动提升注入优先级权重,提升幅度与出现频率正相关,权重上限可配置。
- AC3:语义重复的意图条目自动合并为一条,合并策略:保留正例/负例集合的并集,内容取最完整版本,来源引用全部保留。
- AC4:长期未被触发(连续 N 天未命中注入,N 可配置)的意图条目自动降低优先级权重;超过衰减周期仍未触发的条目直接删除。
- AC5:每次治理运行后生成治理报告,包含:新识别高频主题数、合并条目数、权重提升条目数、衰减条目数、标记待清理条目数。报告持久化存储,可在工作台查看历史。
- AC6:治理频率、聚类相似度阈值、衰减周期、权重提升系数均通过配置文件调整,支持运行时热更新。
- AC7:治理操作全部可回退--合并可拆分还原,权重调整可手动覆盖,待清理标记可撤销。

### 域 F:规则订阅与分发(Rule Subscription & Distribution)

负责统一管理系统规则,按需分发给各 Agent。

#### FR-F01:规则注册与管理

统一注册和管理系统规则。

- AC1:规则条目包含规则内容、适用范围、优先级、生效时间和失效条件。
- AC2:规则支持版本管理,变更历史可追溯。
- AC3:规则之间的优先级和覆盖关系被显式管理。

#### FR-F02:订阅机制

Agent 按需订阅相关规则。

- AC1:Agent 可按角色、域、场景订阅规则集。
- AC2:订阅关系变更时,系统自动调整分发范围。
- AC3:Agent 可查询自己当前订阅的规则清单。

#### FR-F03:规则分发

规则变更时按订阅关系推送给相关 Agent。

- AC1:规则变更后,Agent 在下一次启动或主动查询时获取最新版本;高优先级规则变更主动通知订阅者。
- AC2:分发记录可追溯,能确认哪些 Agent 已收到哪个版本的规则。
- AC3:分发失败时自动重试,超过重试次数后报警。

### 域 G:知识图谱与洞察(Knowledge Graph & Insights)

基于知识条目之间的关联关系构建知识图谱,提供结构化洞察和可视化探索能力。这是 KIVO 从"存储知识"到"理解知识结构"的关键跃迁。

#### FR-G01:图谱构建

基于 Knowledge Entry 的关联关系自动构建和维护知识图谱。

- AC1:图谱节点对应 Knowledge Entry,边对应关联关系(supplements / supersedes / conflicts / depends_on)。
- AC2:图谱支持多种关联信号源:显式关联(FR-B03 维护的关联关系)、来源共现(来自同一文档或对话的条目)、语义邻近(Embedding 相似度超过阈值的条目)。
- AC3:新知识入库或关联关系变更时,图谱增量更新,不需要全量重建。
- AC4:图谱数据通过 API 暴露,支持 Workbench 可视化和外部消费。

#### FR-G02:图谱洞察

基于图谱结构自动发现知识组织中的模式和问题。

- AC1:识别孤立节点--已入库但几乎不参与任何关联的知识条目,提示可能的关联缺失或低价值条目。
- AC2:识别桥接节点--连接多个主题群的关键知识条目,标注其重要性等级。
- AC3:识别稀疏社区--条目数量不少但互相支撑关系弱的主题群,提示需要补充关联或深化研究。
- AC4:发现意外关联--跨主题的非显然关联,基于 Adamic-Adar 共同邻居算法或类型亲和度计算。
- AC5:洞察结果可一键生成调研任务(与域 D 联动),用户可从洞察直接触发知识补齐。

#### FR-G03:图谱可视化

在 Workbench 中提供交互式知识图谱浏览。

- AC1:图谱以力导向布局展示,节点按知识类型着色,边按关联类型区分样式。
- AC2:支持缩放、平移、节点拖拽和聚焦(点击节点高亮其直接关联)。
- AC3:支持按知识域、知识类型、时间范围过滤图谱展示范围。
- AC4:节点点击展示知识条目预览卡片,支持跳转到条目详情页。
- AC5:洞察结果(孤立节点、桥接节点、稀疏社区)在图谱上以视觉标记高亮。

#### FR-G04:知识图谱自动对齐

entries 表发生变更(新增/修改/删除)后,图谱通过定时任务自动增量更新,确保 graph_nodes 和 graph_edges 与 entries 保持一致。与 FR-G01 AC3 的事件驱动增量更新互补--cron 作为兜底机制,覆盖事件遗漏或批量导入场景。

- AC1:系统配置 cron 定时任务,每天执行一次图谱增量构建(`kivo graph build` 或等效增量命令),调度时间排在知识提取 cron(FR-A01 session 提取)之后。
- AC2:增量构建只处理上次构建以来发生变更的 entries(基于 updated_at 时间戳或变更标记),不全量重建。
- AC3:新增 entry 后,下一个 cron 周期内 graph_nodes 自动包含该 entry 对应的节点。
- AC4:删除 entry 后,下一个 cron 周期内对应的 graph_nodes 和关联 graph_edges 自动清理。
- AC5:cron 执行结果写入日志,包含本次处理的 entries 数量、新增/删除的节点和边数量、执行耗时。异常时输出错误信息但不中断后续 cron 任务。
- AC6:`kivo init` 自动注册图谱重建定时任务到 OpenClaw 内置调度(写入 cron 配置或等效调度机制),用户无需手动配置 crontab。
- AC7:`kivo init` 注册调度前检查是否已存在图谱重建任务(幂等性),已存在时跳过注册并输出提示,避免重复注册。
- AC8:图谱重建频率通过配置文件 `graph.rebuildSchedule` 字段管理,默认值为每天一次(`0 3 * * *`),用户可修改为任意 cron 表达式。`kivo init` 注册调度时读取该配置。

优先级:P1

#### FR-G05:知识注入图谱关联增强

向量检索命中 top N entries 后,通过图谱边关系扩展关联 entries,使知识注入不仅覆盖语义相似的条目,还覆盖结构关联的条目。

- AC1:知识注入流程在向量检索(FR-E01)返回 top 10 结果后,对每个命中的 entry 查询 graph_edges,找到 weight 超过配置阈值的强关联 entries。
- AC2:图谱扩展的 entries 与向量检索结果去重后合并,数量由权重阈值动态决定--超过阈值的全部返回,低于阈值的不返回。设置防爆上限(默认 20)防止极端情况撑爆 context,但正常情况下数量完全按需。禁止硬编码固定返回数量。
- AC3:图谱扩展的 entries 按 edge weight 降序排列,优先注入关联强度最高的条目。
- AC4:注入结果中标注每条 entry 的来源路径(vector_match 或 graph_expansion),便于调试和可观测。
- AC5:图谱扩展的 weight 阈值可通过配置调整,默认值基于当前图谱 edge weight 分布的 P75 值。
- AC6:当图谱数据不可用(graph_edges 为空或查询失败)时,降级为纯向量检索结果,不阻断注入流程。
- AC7:`kivo init` 自动将包含图谱关联增强逻辑的知识注入 hook 安装到 OpenClaw hooks 目录。hook 文件为薄壳(符合 §9.1 薄壳约束),核心逻辑通过 `require('@self-evolving-harness/kivo')` 调用包 API。
- AC8:安装的 hook 同时覆盖向量检索注入和图谱扩展注入,用户无需分别配置两个 hook。
- AC9:`kivo init` 安装 hook 前检查目标目录是否已存在同名 hook(幂等性),已存在时对比版本--版本更新则覆盖并提示,版本相同则跳过。

优先级:P1

### 域 H:系统词典(System Dictionary)

负责统一管理术语定义,为 Agent 提供术语约束注入,确保全系统术语语义一致。术语条目是 KnowledgeEntry 的一种特化视图,复用已有的存储、检索、冲突检测和生命周期基础设施。术语的主要来源是 KIVO 知识提取管线从会话中自动提取的专有名词和领域术语,手动录入为补充。

#### FR-H01:术语注册与存储

- AC1:术语条目包含:术语名(term)、英文缩写(abbreviation)、英文全称(fullName)、中文名(chineseName)、定义(definition)、语义约束(constraints)、正例(positiveExamples)、负例(negativeExamples)、适用域(scope)、别名(aliases)。其中 abbreviation、fullName、chineseName 为必填字段。
- AC2:术语条目以 KnowledgeEntry 形式存储,type 为 fact,domain 为 system-dictionary,结构化字段存放在 metadata 扩展中。
- AC3:术语名在同一 scope 内唯一,重复注册时拒绝并提示已有条目。
- AC4:术语检索支持精确匹配模式--术语名或别名完全匹配时直接返回,优先级高于语义相似度排序。
- AC5:术语的唯一性校验、冲突检测、格式验证均在 API 层执行。

#### FR-H02:术语 Prompt 注入

Agent 执行任务时,自动将相关术语约束注入 prompt 上下文。

- AC1:注入格式包含术语名、定义、约束、正例和负例。
- AC2:注入时按任务上下文的语义相关度筛选术语(向量语义检索),不全量灌注。禁止使用 FTS、关键词匹配、正则表达式筛选术语。术语搜索的相关性评分必须基于向量余弦相似度,禁止关键词命中率评分。term-search 组件的评分逻辑只允许使用 BGE embedding cosine similarity。
- AC3:术语注入的 token 开销纳入总 token 预算管理,术语注入优先级高于一般知识注入。
- AC4:注入内容支持 markdown 和 plain text 两种格式。
- AC5:已废弃的术语直接从知识库删除;删除前自动创建替代术语条目(包含旧术语名作为别名),确保 prompt 中出现旧术语名时能通过替代条目注入迁移指引。

#### FR-H03:术语冲突检测

- AC1:冲突类型包括:语义矛盾、范围重叠(scope 交集内 constraints 不兼容)、别名冲突。
- AC2:语义矛盾检测复用冲突检测管线(embedding 粗筛 + LLM 精判),术语域的粗筛阈值可独立配置。
- AC3:别名冲突检测为本地精确匹配,不依赖 LLM。
- AC4:冲突检测结果包含冲突类型、冲突对和建议解决方式。

#### FR-H04:术语生命周期

- AC1:术语生命周期操作包括:新增、修改、废弃、合并。
- AC2:definition 或 constraints 字段变更触发新版本;aliases、examples、scope 变更为 metadata 更新,不触发版本递增。
- AC3:废弃术语复用知识过期与清理流程(FR-C02)。
- AC4:合并术语时,被合并条目状态变为 superseded 并指向合并后的条目,操作可回退。
- AC5:术语变更生成变更事件,推送给订阅了该术语所属 scope 的 Agent。

#### FR-H05:术语来源与批量导入

- AC1:支持从 JSON、YAML、CSV 格式文件批量导入术语,导入时逐条执行冲突检测。
- AC2:支持从治理文件中解析术语定义并导入,作为文档知识提取(FR-A02)的一种特化策略。
- AC3:批量导入生成导入报告,记录成功条数、冲突条数、跳过条数和失败原因。
- AC4:支持批量导出为 JSON/YAML/CSV,导出格式与导入格式对称。
- AC5:知识提取管线(域 A)在对话提取过程中自动识别专有名词和领域术语,符合术语结构的候选自动写入系统词典。这是系统词典的主要内容来源,手动录入为补充。

### 域 I:OpenClaw 集成层(OpenClaw Integration)

负责 KIVO 引擎与 OpenClaw 宿主环境之间的能力声明、运行时集成和插件对接。

#### FR-I01:OpenClaw 能力声明与注册

- AC1:OpenClaw 通过集成接口声明自身能力(LLM 调用、网络访问、文件读写、工具执行),KIVO 据此决定可激活的功能域。
- AC2:能力声明包含能力标识、版本约束和调用约定。
- AC3:OpenClaw 能力在运行期间变更时,KIVO 收到通知并降级受影响的功能,不中断整体服务。

#### FR-I02:LLM Provider 管理

- AC1:支持注册多个 LLM Provider,每个 Provider 声明支持的能力(文本生成、Embedding 生成、结构化输出)。
- AC2:KIVO 按任务类型选择匹配的 Provider,选择策略可配置。
- AC3:当前 Provider 不可用时自动切换到备选 Provider;无可用 Provider 时依赖 LLM 的功能进入降级状态。

#### FR-I03:意图管线 OpenClaw 集成

KIVO 通过 OpenClaw 的插件/扩展机制将意图知识管线接入运行时,实现对话流自动采集和知识自动注入。

- AC1:KIVO 提供插件接口,通过注册 OpenClaw 生命周期钩子(对话结束、prompt 构建前、任务派发前)接入意图管线。
- AC2:插件采用 fail-open 设计--插件加载失败或 hook 执行异常时,OpenClaw 核心功能不受影响,错误仅记录日志。
- AC3:插件提供一键安装命令,自动注册到 OpenClaw 配置、写入默认参数,用户只需重启 OpenClaw 即可生效。
- AC4:插件配置支持热更新--修改阈值、token 预算等参数后无需重启,下次 hook 触发时自动读取新配置。

#### FR-I05:定时任务 CLI 管理(Cron Task Management)

提供 CLI 工具管理 KIVO 的定时任务(图谱重建、知识提取等),支持查看、修改调度频率和手动触发。

- AC1:提供 CLI 命令 `kivo cron install`,将图谱重建任务注册到系统 crontab(或用户级 crontab),频率读取配置文件 `graph.rebuildSchedule`(默认每天一次)。
- AC2:`kivo cron install` 支持 `--schedule <cron-expression>` 参数覆盖配置文件中的频率设置。
- AC3:`kivo cron install` 执行前检查是否已存在同名 cron 条目(幂等性),已存在时更新频率而非重复添加。
- AC4:提供 `kivo cron uninstall` 命令,清除已注册的 KIVO 相关 cron 条目。
- AC5:`kivo cron status` 输出当前已注册的 KIVO cron 任务列表、下次执行时间和上次执行结果。
- AC6:注册的 cron 条目包含标识注释(如 `# KIVO: graph-rebuild`),便于识别和管理。

优先级:P1

### 域 K:知识管线编排(Knowledge Pipeline Orchestration)

#### FR-K01:管线阶段编排

- AC1:对话提取管线精简为五步:原子分解+去上下文化(一步完成) → 行为变化测试+三重测试(准入门禁 FR-A07) → 0.92 cosine 去重 → 素材暂存(staging) → 抽象聚合(产出知识入库)。文档提取管线保留完整阶段(提取 → 分析中间产物生成 → 分类 → 冲突检测 → 合并检测 → 入库)。
- AC2:阶段之间通过事件驱动流转,支持异步执行。
- AC3:任一阶段失败时,管线中断并记录失败上下文,不影响其他独立管线实例。
- AC4:管线支持扩展--新增处理阶段只需注册到管线,不修改已有阶段逻辑。

#### FR-K02:知识分类与路由

- AC1:分类器识别知识条目的类型和所属知识域。
- AC2:分类结果决定后续处理路径:不同知识类型可应用不同的冲突检测策略和入库规则。
- AC3:分类置信度低于阈值时,条目标记为待人工确认,不自动进入后续管线阶段。

### 域 L:分析中间产物(Analysis Artifacts)

负责管理知识提取过程中产生的结构化分析结果。提取管线拆成"分析 → 生成"两步,先把语义中间层做扎实,再进入内容生成,提升提取稳定性和可审计性。

#### FR-L01:分析产物生成

知识提取的第一步产出结构化分析结果,而非直接生成知识条目。

- AC1:分析产物(Analysis Artifact)包含以下结构化字段:sourceId、extractedClaims(提取的断言列表)、entityCandidates(实体候选)、conceptCandidates(概念候选)、linkCandidates(关联建议)、conflictCandidates(冲突建议)、gapCandidates(缺口建议)、reviewCandidates(需人工审核的事项)、recommendedResearchQueries(推荐调研问题)。
- AC2:分析产物与来源文档/对话关联,支持溯源。
- AC3:分析产物持久化存储,不随知识条目生成后丢弃。

#### FR-L02:分析产物审核与消费

分析产物作为知识生成的输入,支持人工审核和自动消费。

- AC1:Workbench 提供分析产物浏览界面,用户可查看每次提取的分析中间结果。
- AC2:用户可对分析产物中的候选项逐条确认、拒绝或编辑,确认后的候选项进入知识生成阶段。
- AC3:自动模式下,高置信度的分析产物直接进入知识生成,低置信度的进入审核队列。
- AC4:分析产物中的 gapCandidates 和 recommendedResearchQueries 可一键创建调研任务(与域 D 联动)。

### 域 M:知识域目标声明(Domain Purpose)

负责为每个知识域定义目标、关键问题和研究边界,指导知识采集和治理方向。解决"知识该不该入库、如何分类、是否值得继续调研"的判断依据问题。

#### FR-M01:域目标配置

每个知识域可配置目标声明,约束该域的知识采集和治理行为。

- AC1:域目标声明包含以下字段:domainId、purpose(目标描述)、keyQuestions(关键问题列表)、nonGoals(非目标列表)、researchBoundary(研究边界描述)、prioritySignals(高优先级信号列表)。
- AC2:域目标声明通过配置文件或 Workbench 界面管理,支持创建、编辑和删除。
- AC3:域目标声明变更时生成变更事件,通知依赖该域的模块重新评估。

#### FR-M02:域目标驱动的行为约束

域目标声明影响知识提取、检索和调研的行为。

- AC1:知识提取时,系统参考目标域的 purpose 和 researchBoundary 判断提取内容是否在域范围内,超出范围的内容标记为"域外"不自动入库。
- AC2:语义检索时,检索排序考虑条目与域目标的相关度--与 keyQuestions 相关的条目排序权重更高。
- AC3:缺口检测时,系统参考 keyQuestions 评估哪些问题尚未被知识库覆盖,生成针对性的调研建议。
- AC4:调研任务生成时,researchBoundary 作为调研范围约束注入任务定义,防止调研方向发散。
- AC5:域目标与知识条目的相关度匹配必须使用向量余弦相似度(BGE embedding),禁止使用关键词匹配或正则表达式做域目标约束判断。domain-goal-constraints 组件的匹配逻辑只允许使用向量语义检索。

### 域 N:知识治理(Knowledge Governance)

负责知识库的持续治理:准入门禁、语义去重、质量审计、从 badcase 中自动学习、定期存量治理。确保知识库不膨胀、不退化、持续进化。

#### FR-N01:知识 MECE 治理

新增知识前自动检测语义重叠,定期扫描去重,审计知识覆盖度盲区。

- AC1:新知识入库前,系统自动用 BGE 向量匹配已有知识条目,语义相似度超过可配置阈值(默认 0.85)时标记为疑似重复,阻断自动入库并提示合并。
- AC2:CLI 命令 `kivo deduplicate` 可用,对全库执行语义去重扫描,输出去重报告(重复对列表、相似度分数、合并建议)。
- AC3:去重报告中每对重复条目附带合并建议(保留哪条、合并策略、合并后内容预览),用户可一键确认合并或忽略。
- AC4:`kivo deduplicate --auto` 模式下,相似度超过高阈值(默认 0.95)的重复对自动合并,合并操作可回退。
- AC5:覆盖度审计--系统基于 Domain Purpose(域 M)的 keyQuestions 检查每个知识域是否存在知识盲区,输出覆盖度报告(已覆盖问题 / 未覆盖问题 / 覆盖率百分比)。
- AC6:语义相似度检测必须使用 BGE 向量 embedding,禁止关键词匹配、正则表达式或规则引擎。
- AC7:`kivo deduplicate --domain <value>` 支持按知识域限定扫描范围,减少大库扫描耗时。

#### FR-N02:Badcase→意图知识自动转化

从用户纠偏、审计发现、复验失败等 badcase 中自动提取真实意图,结构化写入知识库。

- AC1:系统识别三类 badcase 来源:用户对话中的纠偏(correction 信号)、审计报告中的问题发现、任务复验失败记录。
- AC2:从 badcase 中通过 LLM 语义提取真实意图,输出结构化意图知识:场景描述、触发条件、预期行为、反模式(用户不想要的行为)。
- AC3:提取的意图知识以 Knowledge Entry(type=intent)形式入库,走标准知识管线(冲突检测 → 去重检测 → 入库),自动关联已有相关知识条目。
- AC4:每条从 badcase 转化的知识条目自动标注来源类型(user_correction / audit_finding / verification_failure)和来源日期。
- AC5:CLI 命令 `kivo learn-from-badcase` 可用,支持 `--source <path>` 指定 badcase 文件或目录,支持 `--dry-run` 预览提取结果不写入。
- AC6:入库前自动与已有意图知识做语义去重(复用 FR-N01 的 BGE 向量匹配),重复时合并而非新增。
- AC7:意图提取必须使用 LLM 语义理解,禁止规则引擎或模板匹配。
- AC8:系统性问题自动识别--当语义相似度超过可配置阈值(默认 0.85,复用 FR-N01 的 BGE 向量匹配)的一组纠偏记录在 N 天内(N 可配置,默认 14 天)累计超过 M 次(M 可配置,默认 3 次),系统自动将其标记为「系统性问题」而非意图知识。判定依据:该规则属于 LLM 通用能力范围内的常识但 Agent 反复违反,说明根因在流水线或架构层面,一条意图知识无法解决。系统生成结构化问题报告(问题描述、纠偏频率、影响范围、建议修复层级),路由到 SEVO spec 待办而非 KIVO 意图库。
- AC9:系统性问题报告包含路由建议:应修改 SEVO 流水线的哪个阶段,或应优化 KIVO 架构的哪个模块。

#### FR-N03:知识质量审计

审计知识库中每条知识的内容质量,标记不合格条目并生成改写建议。

- AC1:CLI 命令 `kivo audit-quality` 可用,对全库或指定域的知识条目逐条评估内容质量,评估维度:具体性(是否有明确场景)、可执行性(是否能直接指导行动)、场景化(是否绑定触发条件)。
- AC2:每条知识条目输出质量评分(1-5 分),评分由 LLM 基于三个维度综合判定。评分标准:5=三维度全满足;4=两维度满足;3=一维度满足;2=模糊但有信息量;1=空泛无法指导行动。
- AC3:评分 ≤ 2 的条目标记为「质量不合格」,系统为每条不合格条目生成改写建议(保留原意的前提下提升具体性和可执行性)。
- AC4:CLI 命令 `kivo upgrade-quality --batch` 可用,对所有质量不合格条目批量执行 LLM 改写,改写结果以新版本入库(旧版本保留为 superseded),改写操作可回退。
- AC5:`kivo audit-quality --domain <value>` 支持按知识域限定审计范围。
- AC6:审计完成后输出质量报告:总条目数、各评分段分布、不合格率、Top-10 最差条目列表。
- AC7:质量评估必须使用 LLM 语义判定,禁止规则引擎或关键词匹配。
- AC8:`kivo audit-quality --threshold <N>` 支持自定义不合格阈值(默认 2),低于等于该阈值的条目标记为不合格。

#### FR-N04:知识入库价值门禁(Knowledge Intake Value Gate)

入库前评估知识是否填补 LLM 能力与用户预期之间的差距,拦截低价值知识,确保知识库只沉淀 LLM 自身无法覆盖的高价值知识。

核心判定原则:知识条目的价值 = 填补 LLM 能力与用户预期之间的差距。

高价值知识(应入库):用户私有术语/黑话(LLM 不可能知道)、反复出现的 badcase(LLM 知道但做不到)、专业领域定制意图(LLM 理解浅)、容易被 context 稀释遗忘的关键约束、复杂/隐含/间接意图(LLM 容易误判)、用户特有的指代和简写。

低价值知识(不应入库):通用常识(LLM 本来就会)、被反复纠偏的通用规则(说明是系统性问题,该改流水线或架构,不该加知识条目)。

- AC1:知识条目入库前,系统通过 LLM 评估该条目是否填补 LLM 能力盲区,评估维度:私有性(LLM 训练数据是否覆盖)、场景特异性(是否绑定用户特有上下文)、纠偏频率(是否源自反复出现的 badcase)。
- AC2:评估结果为「低价值」的条目阻断入库,直接丢弃并附带拒绝理由(如「属于 LLM 通用能力范围内的常识」)。
- AC3:评估结果为「高价值」的条目正常进入后续管线(冲突检测 → 合并检测 → 入库)。
- AC4:价值评估必须使用 LLM 语义判定,禁止关键词匹配或规则引擎。
- AC5:价值评估阈值可配置,支持运行时热更新。
- AC6:CLI 命令 `kivo audit-value` 可用,对已有知识条目批量执行价值评估,输出报告(高价值/低价值/待定分布、Top-10 低价值条目列表),支持 `--domain <value>` 限定范围。

#### FR-N05:统一入库质量门禁(Unified Intake Quality Gate)

所有知识入库路径(批量萃取、实时提取、CLI 录入、API 写入、文档导入)统一经过质量门禁,确保知识库只接收高质量、不重复、不矛盾的知识条目。FR-N01 提供语义去重能力,FR-N04 提供价值评估能力;FR-N05 将去重、价值评估、冲突检测、过期检测整合为统一的入库前置门禁,所有写入路径必须经过此门禁。

- AC1:向量相似度去重--入库前使用 BGE 向量 embedding 计算候选条目与已有知识的 cosine 相似度,>= 0.92 时判定为重复,重复条目跳过入库(或仅更新已有条目的时间戳)。去重阈值 0.92 通过 kivo.config.json 的 `extraction.dedup.threshold` 配置,支持运行时热更新。去重必须使用 BGE 向量检索,禁止 FTS 或关键词匹配。
- AC2:LLM 价值评估--对候选条目执行价值判定,临时状态信息、纯操作记录、通用常识等低价值内容直接丢弃,不进入后续管线。
- AC3:冲突检测--复用 FR-C01 的冲突检测与解决能力,候选条目与已有知识存在事实矛盾时按 FR-C01 定义的策略自动处理(时间优先、来源优先)。
- AC4:过期检测--候选条目关联的事实发生变化时(如版本号更新、日期过期、状态变更),自动删除关联的旧知识条目,删除前将过期原因写入门禁日志(AC5)。若旧条目有部分仍有效的内容,先提取有效部分生成新条目再删除旧条目。
- AC5:门禁结果持久化--每次门禁执行的判定结果(通过、拒绝、合并、冲突、过期标记)写入数据库,包含候选条目内容、判定类型、判定理由、执行时间,支持审计查询。
- AC6:被拒绝的条目写入 rejected log,不进入主知识库。rejected log 支持按时间、拒绝原因、来源路径查询,用于分析入库质量趋势。
- AC7:门禁对所有入库路径强制生效,不存在绕过门禁的写入方式。CLI 提供 `--no-quality-gate` 参数用于调试场景,生产环境默认不可用。

#### FR-N06:批量萃取质量治理(Batch Extraction Quality Governance)

对 FR-A05 会话历史知识萃取的产出施加质量门禁,确保批量提取不向知识库灌入低质量条目。

- AC1:`kivo extract-sessions` 的产出在写入知识库前,逐条经过统一质量门禁(FR-N05)过滤。
- AC2:质量门禁默认开启。`--no-quality-gate` 参数可关闭门禁(仅用于调试或迁移场景),关闭时输出警告日志。
- AC3:萃取完成后输出门禁统计日志:萃取总条目数、门禁通过数、拒绝数、合并数,以及各拒绝原因的分布。
- AC4:门禁拒绝率超过可配置阈值(默认 80%)时,输出告警提示萃取源质量异常,建议检查 session 筛选条件或噪音过滤规则。

#### FR-N07:知识条目命名规范(Entry Title Convention)

知识条目的 title 字段强制遵守长度规范,确保知识库中所有条目名称简洁可读、适合列表展示和图谱节点标签。content 字段不设长度限制。

- AC1:所有入库路径(批量萃取、实时提取、CLI 录入、API 写入、文档导入)统一校验 title 字段长度不超过 20 字,不合规时阻断入库。
- AC2:超长 title 自动压缩策略按优先级依次执行--去括号内容 → 去冒号后英文注释 → 截断至 20 字。
- AC3:压缩后仍超过 20 字时,截断至 19 字并追加省略号("..."),确保最终长度 ≤ 20 字。
- AC4:content 字段不设长度限制,允许任意长度的知识描述。
- AC5:提供存量治理脚本 `kivo governance title-normalize`,对已入库的超长 title 执行压缩清洗,输出处理统计(总条目数、压缩数、截断数、无需处理数)。

#### FR-N08:知识库定期治理(Periodic Knowledge Governance)

定时自动执行知识库健康维护,检测过时条目、衰减低活跃知识、扫描逻辑矛盾、回收已成为通用常识的条目。准入门禁(FR-N05)管入口质量,定期治理管存量健康--两者共同确保知识库长期不膨胀、不退化。治理的核心动作是聚合 → 抽象 → 规律提炼 → 有效性评估,相似度只是发现候选的手段,不是治理本身。治理的产出是更少、更精、更抽象的条目。

- AC1:定期治理任务由 cron 自动触发,默认每天执行一次,执行频率可配置。不依赖人工启动,系统安装后自动生效。
- AC2:过时检测--LLM 逐条评估知识描述的事实状态是否仍然成立(如版本号已更新、工具已废弃、流程已变更),过时条目直接删除并附带过时原因记录。
- AC3:价值衰减--系统记录每条知识被检索命中的时间戳(最后命中时间),连续 N 天(N 可配置,默认 90 天)未被命中的条目自动标记为低活跃;超过衰减周期(默认 180 天)仍未命中的条目直接删除。
- AC4:矛盾扫描--对全库知识条目分批执行两两逻辑一致性检测(LLM 判定),发现逻辑矛盾的条目对自动执行冲突解决(复用 FR-C01 的冲突解决机制)。分批策略:按知识域分组,每批不超过 500 条,避免单次执行耗时过长。
- AC5:常识回收--LLM 重新评估每条知识是否已成为当前模型的通用能力(随 LLM 迭代升级,之前有价值的私有知识可能变成常识)。判定为常识的条目直接删除,附带删除理由「已被 LLM 通用能力覆盖」。
- AC6:治理结果持久化--每次定期治理的执行结果写入数据库:执行时间、扫描条目数、各操作类型统计(deleted / conflict_detected)、具体操作明细(条目 ID + 操作类型 + 理由)。支持审计查询和趋势分析。
- AC7:删除操作记录删除原因和操作时间,但不保留条目内容。
- AC8:CLI 命令 `kivo governance run` 可手动触发一次完整的定期治理流程,支持 `--domain <value>` 限定扫描范围,支持 `--dry-run` 预览治理结果不执行写入。
- AC9:治理完成后输出治理报告:扫描总条目数、删除数、新发现矛盾数、常识回收数,以及 Top-10 高风险条目列表(即将触达删除阈值的条目)。
- AC10:所有治理判定必须使用 LLM 语义理解,禁止关键词匹配或规则引擎。矛盾扫描的向量预筛选使用 BGE embedding(先用向量找语义相近的候选对,再用 LLM 精判是否矛盾),减少 LLM 调用量。
- AC11:BGE 向量预筛分批处理--矛盾扫描的 BGE embedding 预筛阶段必须分批执行,避免一次性对全量条目调用 embedding 导致超时中断。具体要求:
  - 每批处理不超过 50 条知识条目的 embedding 计算与余弦相似度比对,批次大小可配置。
  - 每批之间设置间隔(默认 500ms,可配置),避免 BGE 服务过载。
  - 单批超时(默认 30s,可配置)不影响后续批次执行:超时批次记录错误日志并跳过,下次治理执行时优先重试失败批次。
  - 扫描进度可恢复:每批完成后将已处理的条目 ID 范围写入检查点(checkpoint),治理任务中断后下次执行从检查点继续,不重复已完成批次。
  - 检查点持久化到数据库,包含:扫描轮次 ID、已完成批次范围、失败批次列表、最后更新时间。
  - 全量扫描完成(所有批次成功或重试耗尽)后清除检查点,下次治理从头开始新轮次。

#### FR-N09:碎片聚合治理(Fragment Aggregation Governance)

定期扫描知识库中的碎片条目,执行聚合 → 抽象 → 规律提炼 → 有效性评估,产出更少、更精、更抽象的高密度知识条目。治理的核心动作是提炼黄金--从碎片中收敛出少量高密度抽象知识,而非简单的相似度去重。

碎片的定义:同主题多条、粒度过细、绑定特定时刻或场景、未经抽象的条目。典型特征:多条条目说的是同一件事的不同侧面,合并后信息密度更高。

- AC1:定期扫描知识库中的碎片条目。碎片识别策略:用 BGE 向量检索找到语义相近的条目簇(作为候选),再用 LLM 精判这些候选是否属于同主题碎片。禁止使用关键词匹配、正则表达式、FTS5、规则引擎承担语义理解职责。
- AC2:对碎片执行 LLM 语义聚合--多条碎片 → 一条抽象知识。聚合的目标是提炼跨场景、跨时间的规律和原则,而非简单拼接或取交集。聚合由 LLM 执行,禁止关键词匹配、正则表达式、FTS5、规则引擎。
- AC3:聚合后的知识条目必须通过 FR-A07 定义的三重测试(时效性、跨场景、抽象性)。不通过的聚合结果不入库,原始碎片保留等待下次治理。
- AC4:聚合完成后删除原始碎片。删除操作记录聚合来源(哪些碎片 ID 被聚合为哪条新知识),支持审计追溯。
- AC5:有效性评估必须是设计过的,不是跑个相似度阈值就完事。评估维度:这条知识在当前环境下还成立吗?有没有被更新的理解 supersede?它指导的行为是否仍然正确?评估由 LLM 执行,禁止关键词/正则/规则引擎。
- AC6:治理频率可配置(默认每天一次,与 FR-N08 定期治理同步执行或独立调度)。CLI 命令 `kivo governance aggregate` 可手动触发,支持 `--domain <value>` 限定范围,支持 `--dry-run` 预览。
- AC7:聚合治理完成后输出报告:扫描总条目数、识别碎片簇数、聚合产出新知识数、删除碎片数、聚合失败(未通过三重测试)数。
- AC8:聚合产出的知识条目必须能回答「它让 agent 在什么场景下避免了什么错误」。回答不了的聚合结果不入库。这是目标结果导向约束--知识库和图谱存在的唯一目的是让 agent 做出更好的判断。
- AC9:禁止硬编码定量。不写死「每次最多聚合 N 条」「相似度阈值 0.X」等魔法数字。聚合的触发和范围由 LLM 语义判断决定。

- AC2:指标同时展示绝对值和趋势变化(较上周)。
- AC3:所有内部链接使用正确的 basePath 前缀。

#### FR-W02:知识列表与搜索

#### FR-W03:知识条目详情

展示知识条目的完整信息。

- AC1:详情页标题区域显示 ≤20 字的短标题,超出截断并显示省略号。

#### FR-W04:活动流

- AC1:活动流记录核心事件:条目创建、更新、废弃、冲突检测、冲突解决、调研完成、规则变更。

#### FR-W06:知识条目操作

用户在 Workbench 中对知识条目执行内容编辑和删除。

- AC1:支持删除过时的知识条目,删除后不保留历史。
- AC2:编辑提交后生成新版本,新版本经过冲突检测管线。
- AC3:编辑操作使用乐观锁(expectedVersion),版本冲突时提示用户刷新后重试。

#### FR-W07:调研管理

调研任务的查看、创建和管控。

- AC1:调研任务列表展示任务状态、优先级、创建时间和预算消耗,支持按状态筛选和分页。
- AC2:缺口报告列表展示盲区主题、影响面和填补进度。

#### FR-W16:知识图谱导航与稳定性

### 域 X:访问控制与可观测性(Access Control & Observability)

- AC2:完成任一入口操作后,系统引导用户前往知识库查看结果并尝试检索。
- AC3:用户首次搜索无结果时,给出下一步操作建议。
- AC4:主界面展示系统就绪度检查清单。

#### FR-Z07:错误提示与恢复动作

- AC1:覆盖用户操作全链路的常见失败场景,每类场景有用户可读的错误提示。
- AC2:错误提示包含原因说明和建议的下一步动作。
- AC3:可重试的失败支持一键重试。
- AC4:关键失败场景支持复制诊断信息。

#### FR-Z08:认证生命周期

- AC1:单用户密码认证,通过 bootstrap 流程设置密码。
- AC2:支持密码修改、会话超时自动失效和主动退出。
- AC3:操作审计覆盖知识生命周期关键动作。

#### FR-Z09:文档与实现一致性门禁

- AC1:README 和 Quick Start 中所有示例代码纳入 CI 校验,示例不可运行时构建失败。
- AC2:公开 API 签名变更时,文档和示例必须同步更新。

#### FR-Z10:升级与数据迁移

从旧版本升级到新版本时,已有数据安全迁移,不丢失、不损坏。

- AC1:每次 schema 变更附带迁移脚本,支持从上一个正式版本自动迁移到当前版本。
- AC2:升级前自动执行数据完整性校验,校验不通过时中止升级并输出诊断信息。
- AC3:升级失败时支持回滚到升级前状态,回滚操作不丢失升级前的数据。
- AC4:升级说明文档记录每个版本的 breaking changes、迁移步骤和已知兼容性问题。

#### FR-Z11:运行时 Embedding Provider 健康检查

CLI 命令执行前自动检测 embedding provider 连通性,确保向量检索能力可用。

- AC1:所有依赖向量检索的 CLI 命令(query、search、extract-sessions、import 等)执行前,自动检测 embedding provider 连通性。
- AC2:检测内容:endpoint 可达(HTTP 连通)、模型响应正常(发送测试向量请求)。检测超时阈值可配置,默认 5 秒。
- AC3:不可用时输出友好错误信息 + 修复命令,不输出 stack trace。错误信息示例:"KIVO 需要可用的 embedding provider 提供向量检索能力。当前配置:[provider] [endpoint]。请检查服务是否运行:[对应修复命令]"。
- AC4:对于 Ollama 类本地 provider,修复命令示例:`ollama serve`(服务未启动)、`ollama pull <model>`(模型未加载)。对于远程 API 类 provider,提示检查网络和 API key。
- AC5:健康检查结果缓存 60 秒,避免同一会话内重复检测。缓存失效后下次命令重新检测。
- AC6:提供 `kivo doctor` 命令,输出完整的环境诊断报告(Node 版本、存储状态、embedding provider 状态、LLM provider 状态),用于故障排查。

### 域 P:领域知识库(LLM Wiki)

负责管理用户定义的领域知识库(如"线性代数""数理统计""产品设计"等学科/领域),支持结构化知识条目的归类、浏览和语义检索。领域知识和意图知识共享同一个 entries 表(通过 domain 标签区分),因此领域知识天然作为节点纳入知识图谱(域 G)。

#### FR-P01:领域知识库管理

用户可以创建、编辑、删除领域知识库。

- AC1:支持创建领域知识库,每个库包含名称、描述和知识范围定义。名称全局唯一。
- AC2:支持编辑领域知识库的名称、描述和范围定义。
- AC3:支持删除领域知识库,删除前提示该库下的知识条目数量,确认后执行。删除库不自动删除其下的知识条目(条目变为未分类)。
- AC4:领域知识库列表页展示每个库的名称、描述、条目数量和最近更新时间。

#### FR-P02:领域知识条目

每个领域知识库下包含结构化的知识条目,支持 LLM 语义检索。

- AC1:领域知识条目复用 KnowledgeEntry 数据结构,通过 domain 字段归属到对应的领域知识库。
- AC2:领域知识库详情页展示该库下所有条目,支持按类型筛选、分页和语义搜索。
- AC3:支持在领域知识库内手动创建、编辑、删除知识条目。
- AC4:领域知识的检索必须使用向量语义检索(BGE embedding cosine similarity),禁止关键词匹配。

#### FR-P03:领域知识与图谱集成

领域知识条目作为节点纳入知识图谱(域 G),与意图知识共享同一个图谱。

- AC1:领域知识条目入库时自动作为节点加入知识图谱,复用域 G 的图谱更新机制。
- AC2:图谱中可以看到领域知识与意图知识之间的关联关系。
- AC3:图谱节点支持按 domain 筛选,可只查看某个领域知识库的子图。

#### FR-P04:领域知识自动提取

从会话、文档等来源自动提取领域知识并归入对应领域库。

- AC1:知识提取管线(域 A)提取的知识条目,根据内容语义自动匹配到最相关的领域知识库(基于领域库的范围定义做向量匹配)。
- AC2:匹配置信度低于阈值时,条目归入默认域(未分类),用户可后续手动归类。
- AC3:用户导入文档时可指定目标领域知识库,提取结果直接归入该库。
- AC4:领域匹配必须使用向量语义检索,禁止关键词匹配或规则引擎。

## 5. 非功能需求(NFR)

### 5.1 性能

- NFR-5.1:知识检索响应时间 ≤ 2 秒(P95)。
- NFR-5.2:知识提取不阻塞 Agent 主任务执行,异步完成。
- NFR-5.3:规则分发延迟 ≤ 30 秒(从规则变更到订阅者可查询)。
- NFR-5.4:图谱增量更新延迟 ≤ 5 秒(从知识入库到图谱节点/边可查询)。
- NFR-5.5:Workbench 页面首屏加载时间 ≤ 3 秒(P95,标准网络环境)。

### 5.2 可靠性

- NFR-5.6:知识条目写入后不丢失,支持持久化存储。
- NFR-5.7:冲突检测覆盖所有写入路径,不存在绕过冲突检测的入库方式。
- NFR-5.8:调研任务失败不影响知识库已有数据的完整性。
- NFR-5.9:Analysis Artifact 持久化存储,不因管线后续阶段失败而丢失。

### 5.3 可扩展性

- NFR-5.10:知识类型可扩展,新增类型不修改已有提取和存储逻辑。
- NFR-5.11:调研信息源可扩展,新增来源只需实现统一的获取接口。
- NFR-5.12:规则分发机制与具体 Agent 运行时解耦。
- NFR-5.13:图谱关联信号源可扩展,新增信号类型不修改图谱核心计算逻辑。

### 5.4 安全性

- NFR-5.14:知识条目支持访问控制,敏感知识不跨域泄露。
- NFR-5.15:调研任务不访问用户未授权的信息源。
- NFR-5.16:规则分发不包含超出订阅范围的内容。
- NFR-5.17:用户上传的文档在提取完成后不保留原始文件(可配置保留策略)。

### 5.5 OpenClaw 集成约束

- NFR-5.18:KIVO 核心逻辑不硬编码 OpenClaw 的内部路径、凭据或私有 API 细节,通过集成接口层对接。
- NFR-5.19:更换底层存储或检索引擎后,上层接口行为不变。
- NFR-5.20:OpenClaw 集成接口稳定,OpenClaw 升级不要求 KIVO 核心同步修改。

### 5.6 Web 工作台

- NFR-5.21:Web 工作台所有内部链接使用 basePath 前缀,不产生路径错误导致的 404。
- NFR-5.22:空状态页面提供功能说明和引导操作,不展示纯空白。
- NFR-5.23:图谱可视化在 1000 节点规模下保持流畅交互(≥ 30fps)。
- NFR-5.24:搜索结果返回 < 300ms(P95),列表筛选切换 < 200ms,图谱节点点击到详情面板 < 200ms。
- NFR-5.25:所有写操作 < 100ms 出现 loading 态,操作完成反馈 < 2 秒。
- NFR-5.26:新用户首次体验引导 ≤ 5 分钟完成,引导步骤 ≤ 5 步。
- NFR-5.27:所有状态值和类型值本地化展示,不暴露英文枚举原始值。

### 5.7 NFR 验收标准

- AC-5.1:知识检索在 1000 条知识库规模下满足性能指标。
- AC-5.2:模拟冲突场景下,冲突检测无遗漏。
- AC-5.3:更换底层存储或检索引擎后,上层接口行为不变。
- AC-5.4:审计日志能追溯任一知识条目的完整生命周期。
- AC-5.5:图谱可视化在 1000 节点、3000 边规模下交互流畅。
- AC-5.6:搜索和筛选操作在 1000 条知识库规模下满足 NFR-5.24 延迟指标。
- AC-5.7:新用户按引导流程操作,5 分钟内完成首条知识入库并检索命中。
- AC-5.8:在无 Ollama、无 BGE、无网络的环境下执行 `vitest run`,全部测试用例通过且无超时。
- AC-5.9:RelevanceScorer、ContextInjector 等涉及 embedding 的测试,均通过注入 MockEmbeddingProvider 隔离外部依赖。

### 5.8 语义理解技术约束(全局强制)

适用于 KIVO 全部模块,无例外。

- NFR-5.28:禁止使用 FTS5 MATCH、关键词匹配、正则表达式做知识检索、语义理解、相关性评分、去重判断。适用场景包括但不限于:知识检索、相关性评分、语义去重、词典搜索、域目标匹配、意图匹配、注入筛选。
- NFR-5.29:所有涉及语义理解的场景必须使用向量余弦相似度(embedding cosine similarity)。Embedding provider 是系统强制依赖,不存在「embedding 不可用时降级为关键词」的合法路径。provider 不可用时系统报错并引导修复,不降级。
- NFR-5.30:FTS5 表可保留用于精确文本定位(如按 ID 查找、按术语名精确匹配),但禁止用于语义检索、相关性排序或去重判断。
- NFR-5.31:禁止在代码中实现任何形式的「向量不可用时降级为关键词/FTS5」逻辑。向量检索是唯一合法的语义理解路径。Embedding provider 不可用时,系统必须抛出明确错误并附带修复引导(provider 配置方法、服务启动命令、模型拉取命令),禁止静默降级或 graceful skip。

### 5.9 部署态架构约束(全局强制)

适用于 KIVO 所有 OpenClaw 集成组件(hook、cron、CLI wrapper),无例外。

- NFR-5.32:所有 OpenClaw 集成组件必须是薄壳(thin shell)--只做事件监听、参数传递和结果路由,核心逻辑必须调用 KIVO 包暴露的公开 API。禁止集成组件内维护独立的业务逻辑实现。
- NFR-5.33:搜索逻辑、提取逻辑、治理逻辑、注入逻辑在 KIVO 包内只维护一份实现。所有消费方(hook/cron/CLI/Web API)通过 import 包 API 使用,禁止复制代码或重新实现。
- NFR-5.34:`kivo init` 生成的集成文件通过模块引用(`require`/`import`)消费包能力,用户升级包版本后集成文件自动使用新逻辑,无需手动同步。
- NFR-5.35:KIVO 包的公开 API 遵循语义化版本(SemVer),minor/patch 版本保持向后兼容,破坏性变更仅在 major bump 时允许且必须提供迁移指引。

### 5.10 测试独立性(全局强制)

适用于 KIVO 全部自动化测试,无例外。

- NFR-5.36:所有单元测试和集成测试必须在无外部服务(Ollama、BGE、向量数据库等)的环境下通过。测试不得依赖网络连接或本地运行的第三方服务。
- NFR-5.37:涉及外部服务调用的模块,其构造函数或工厂方法必须支持依赖注入。测试通过注入 mock 实现来隔离外部依赖,禁止在测试中触发真实网络请求。
- NFR-5.38:当模块构造函数提供默认实现(如默认连接 localhost 的 embedding provider),测试必须显式注入 mock,不得依赖默认实现的可用性。
- NFR-5.39:测试套件在 CI 环境(无 GPU、无 Ollama、无外部 API)下全量通过,单个测试用例执行时间 ≤ 5 秒。超时即视为外部依赖泄漏。

## 6. 概念架构

### 6.1 系统管什么

KIVO 管理知识的完整生命周期:采集 → 分析 → 结构化 → 存储 → 关联 → 检索 → 注入 → 迭代 → 调研 → 回流。

两个消费面:
- Agent 消费面:通过 API 检索知识、注入上下文、订阅规则。
- 用户消费面:通过 Workbench 导入知识、探索图谱、管理调研、治理知识资产。

### 6.2 管线流转

知识管线(核心路径):

```
信息来源(对话 / 文档 / 规则文件 / URL 抓取 / 手动录入 / 调研结果)
  → 分析中间产物生成(域 L:提取候选实体、概念、关联、冲突、缺口建议)
  → 知识条目生成(基于分析产物,按知识类型分类)
  → 价值门禁(域 N:评估是否填补 LLM 能力盲区,拦截低价值知识)
  → 冲突检测(域 C:与已有知识比对,判定语义矛盾)
  → 冲突解决(自动策略)
  → 合并检测(域 C:无冲突的同主题知识,判定语义互补可合并)
  → 知识入库(持久化、建立关联、更新图谱)
  → 缺口检测(域 D:分析盲区,结合图谱洞察)
  → 调研执行(自主获取信息)
  → 调研结果回流(重新进入分析管线)
```

规则分发管线(独立于知识管线):

```
规则变更检测 → 规则条目更新 → 订阅关系匹配 → 按需推送 → 分发确认
```

意图知识沉淀管线(感知侧):

```
对话结束事件 → 意图信号检测(LLM 语义识别五类信号)
  → 结构化知识提取(标题 / 正文 / 正例 / 负例)
  → 标准知识管线(冲突检测 → 合并检测 → 入库)
  → 持久化向量索引更新
  → 批量补漏(定时扫描历史对话日志,处理遗漏片段)
```

意图增强管线(消费侧):

```
Agent 构建 prompt 前 → 向量匹配相关意图知识 → 价值判定(过滤通用常识,只保留填补 LLM 能力盲区的知识)→ 术语注入 → 上下文注入(token 预算内)→ 意图消歧 → 增强后的请求交给 Agent 处理
派发子任务时 → 向量匹配相关意图知识 → 追加到任务 prompt
```

图谱洞察管线(分析侧):

```
知识入库事件 → 图谱增量更新 → 定期洞察分析(孤立节点 / 桥接节点 / 稀疏社区 / 意外关联)→ 洞察结果写入缺口报告 → 可触发调研任务
```

存量治理管线(定时触发):

```
cron 定时触发(默认每周)
  → 过时检测(LLM 评估事实状态是否仍成立)
  → 价值衰减(最后命中时间超过阈值 → 标记低活跃 → 超衰减周期物理删除)
  → 矛盾扫描(BGE 向量预筛 + LLM 精判,发现矛盾对 → 自动冲突解决)
  → 常识回收(LLM 重新评估是否已被通用能力覆盖 → 物理删除)
  → 治理报告输出 + 结果持久化
```

### 6.3 系统边界

- KIVO 管理知识的提取、存储、迭代、检索、图谱、分发和工作台。
- KIVO 不负责 Agent 的任务执行和流程编排(SEVO 的职责)。
- KIVO 不负责 Agent 效果度量和漂移诊断(AEO 的职责)。
- KIVO 不负责设计产物生成(Claw Design 的职责)。
- KIVO 提供知识检索和规则分发接口,OpenClaw 负责在合适的时机调用。
- KIVO 的调研能力依赖 OpenClaw 提供的信息获取工具,KIVO 定义调研任务,OpenClaw 负责执行。

### 6.4 进化机制

- 提取回路:根据误提取、漏提取和用户纠偏记录提升提取质量。
- 检索回路:根据查询未命中和用户反馈优化检索策略。
- 调研回路:根据调研结果质量和缺口填补效果调整调研策略。
- 规则回路:根据分发失败和规则冲突记录优化规则管理。
- 图谱回路:根据洞察准确度和用户对洞察建议的采纳率优化图谱分析算法。

## 8. 与 Self-Evolving Harness 其他模块的边界

### 8.1 与 SEVO 的边界

SEVO 负责研发流水线的阶段推进和交付闭环。KIVO 为 SEVO 提供知识支撑。

- KIVO 提供:历史规格、方法论、规则、经验、意图线索。
- SEVO 消费:把这些知识用于 Spec 编写、Contract 设计和 Review 参考。
- 边界结论:KIVO 管"知道什么",SEVO 管"如何把一次研发任务做完"。
- Research Task(KIVO)vs Pipeline Task(SEVO):前者填补知识缺口,后者完成研发交付,属于不同层面。

### 8.2 与 AEO 的边界

AEO 负责度量 Agent 效果、发现漂移、诊断根因。KIVO 负责知识资产本身的治理。

- 交互点:AEO 发现效果漂移后,可触发 KIVO 的缺口检测,排查是否因知识缺失导致。
- 边界结论:AEO 管"做得怎么样",KIVO 管"知道什么"。

### 8.3 与 OpenClaw 宿主环境的边界

- OpenClaw 负责:Agent 运行、工具接入、消息调度、执行沙箱。
- KIVO 负责:知识提取逻辑、存储结构、检索算法、冲突解决策略、规则分发协议、图谱计算、工作台 UI。
- 边界结论:KIVO 定义知识管理语义,OpenClaw 做运行时集成。

## 9. 约束与假设

### 9.1 约束

- KIVO 只服务于 OpenClaw 宿主环境,不承诺支持其他平台;但代码不硬编码 OpenClaw 内部路径和私有实现细节,保持工程质量。
- 知识条目的存储格式和检索接口与具体向量数据库解耦。
- 调研任务的执行依赖 OpenClaw 提供的工具能力,KIVO 只定义任务不直接执行。
- 规则分发采用拉取优先策略,推送为补充。
- 图谱计算在本地完成,不依赖外部图数据库(初期)。
- **Hook 薄壳与单一实现约束**:所有 OpenClaw 集成组件(Gateway hook、cron 脚本、CLI wrapper)必须是薄壳,核心逻辑统一由 KIVO 包 API 提供。禁止集成组件内维护独立的搜索、提取、治理实现,确保用户升级包版本后所有运行态组件自动使用新逻辑。具体要求:
  - Gateway hook 只做事件监听、参数解析和结果传递,核心逻辑必须通过 `require('@self-evolving-harness/kivo')` 调用包 API。
  - cron 脚本只做定时触发和参数组装,实际执行逻辑必须调用 KIVO 包的 CLI 命令或编程 API。
  - `kivo init` 生成的 hook 通过 require 引用包模块,`npm update` 后自动使用新版逻辑。
  - KIVO 包公开 API 保持向后兼容,破坏性变更只允许在 major version bump 时发生。
  - hook/cron/CLI 只允许 import `package.json` exports 声明的公开入口,禁止引用包内部路径。
  - 代码审查必须检查 hook/cron/script 中是否存在内联实现,发现则视为 P0 阻断项。
  - `kivo doctor --hooks` 检测已部署 hook 是否引用当前包版本 API 及是否存在内联实现。

### 9.2 假设

- OpenClaw 环境能提供基本的文件读写、语义搜索和网络访问能力。
- Agent 对话记录可被 KIVO 访问用于知识提取。
- 用户接受 Agent 在后台执行调研任务消耗一定计算资源。
- 知识库规模在初期控制在万级条目以内。
- 图谱规模在初期控制在千级节点以内。

### 9.3 非目标

- KIVO 不负责 Agent 的任务执行和流程编排(SEVO 的职责)。
- KIVO 不负责 Agent 效果度量和优化建议(AEO 的职责)。
- KIVO 不负责设计产物生成(Claw Design 的职责)。
- KIVO 不替代通用搜索引擎,调研能力服务于知识缺口填补。
- KIVO 不替代 Obsidian 等通用笔记工具,聚焦结构化知识管理和 Agent 知识消费。
