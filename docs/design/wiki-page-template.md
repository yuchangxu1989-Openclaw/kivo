# KIVO Wiki 页面模板

OpenClaw(ux-01 子Agent)｜2026-05-25

## 1. 结论

一篇 KIVO Wiki 页面必须回答 5 个问题：

1. 这页讲什么。
2. 这页来自哪些材料。
3. 这页包含哪些知识点。
4. 这页和哪些知识相关。
5. 哪些内容是 AI 抽取的，哪些是用户确认或编辑过的。

Wiki 页面不是一段 AI 总结。它是领域知识的阅读入口，必须带来源、关系、批注和可信度线索。

证据：FR-P06 定义 Wiki 页面是从原子知识聚合生成的编译产物；FR-P06 AC5 要求展示来源溯源；FR-P03 AC6 要求知识点详情展示前置知识、相关题目、解题方法、用户批注、易混概念。

## 2. 页面 Wireframe

白底黑字，不使用暗色主题。

```text
┌──────────────────────────────────────────────────────────────┐
│ 面包屑：领域知识库 / 数学 / 概率论与数理统计 / 条件概率        │
│                                                              │
│ 标题：条件概率                                                │
│ 摘要：用一句人话说明这页解决什么问题                          │
│ 标签：概念  重要  来自 3 份材料  12 条知识点  8 条关系          │
│ 状态：AI 生成，用户已编辑 1 次 / 或 AI 生成，未人工确认         │
├──────────────────────────────────────────────────────────────┤
│ 主体正文                                                       │
│ 1. 定义                                                       │
│ 2. 公式                                                       │
│ 3. 适用条件                                                   │
│ 4. 例题                                                       │
│ 5. 常见错误                                                   │
│ 6. 和其他知识的区别                                           │
├──────────────────────────────┬───────────────────────────────┤
│ 来源材料                     │ 关联知识                       │
│ - 文件名 / 页码 / 段落        │ - 前置知识                      │
│ - 原文片段                   │ - 相关题目                      │
│ - 查看原文                   │ - 解题方法                      │
│                              │ - 易混概念                      │
├──────────────────────────────┴───────────────────────────────┤
│ 批注                                                         │
│ - 来自原材料的批注                                           │
│ - 用户在本 Wiki 上新增的批注                                  │
├──────────────────────────────────────────────────────────────┤
│ 图谱位置：mini graph / 前后 1 跳关系                          │
├──────────────────────────────────────────────────────────────┤
│ 编辑历史：最近一次编辑、可查看历史，不默认占主屏               │
└──────────────────────────────────────────────────────────────┘
```

## 3. 标题、摘要、正文

### 3.1 标题

标题必须是用户能理解的知识主题，不是文件名，不是数据库 ID。

好标题：

- 条件概率。
- 全概率公式。
- 贝叶斯公式的后验概率计算。
- 独立事件和互不相容事件的区别。

坏标题：

- Default Space。
- 概率论与数理统计教程配套题库.pdf。
- 提取 61 条知识。
- Untitled。

初始生成：

- 初始标题由 AI 根据聚合主题生成。
- 用户编辑后以用户版本为准。
- 后续自动编译不能覆盖用户手改标题，除非用户选择「重新生成标题」。

证据：陌生人走查记录 Wiki 列表里有大量 PDF 文件名式页面和空壳；FR-A01 AC2 要求标题由 LLM 抽象归纳，不能直接照搬原文。

### 3.2 摘要

摘要回答：这页知识有什么用，什么时候用。

要求：

- 1 到 3 句话。
- 不写「本文介绍了」。
- 不堆术语。
- 能直接帮用户判断是否点进去。

生成方式：AI 生成，用户可编辑。用户编辑后保留人工版本。

### 3.3 正文

正文按知识类型组织。默认模板：

- 定义/核心结论。
- 公式或规则。
- 适用条件。
- 例题或应用。
- 易错点。
- 和相近概念的区别。
- 来源摘录。

如果是方法类页面，模板改为：

- 适用场景。
- 步骤。
- 判断条件。
- 例题。
- 常见错误。

如果是题目类页面，模板改为：

- 题干。
- 考察知识点。
- 解法。
- 答案。
- 易错点。
- 变式。

证据：FR-P02 AC5 定义 concept/method/question/mistake/annotation；FR-P02 AC7/AC8 定义题目和错题扩展字段。

## 4. AI 生成和用户编辑边界

AI 负责：

- 初始标题、摘要、正文结构。
- 从多条 Entry 合并成 Wiki 页面。
- 推荐关联知识。
- 提取来源材料定位。

用户负责：

- 改标题。
- 改正文。
- 删除错误知识。
- 标记不同意。
- 确认或删除关联关系。
- 添加批注。

规则：

- AI 不能静默覆盖用户编辑。
- 自动重新编译时，用户编辑区域保留。
- 页面必须标注内容状态：AI 生成 / 用户已编辑 / 存在争议。

证据：FR-P06 AC3 要求原子知识更新时页面自动重新生成；FR-P07 AC5 允许用户手动干预还原、删除、编辑。两者必须有边界，否则用户手改会被系统覆盖。

## 5. 来源材料

来源材料必须展示。没有来源的 Wiki 页面不可信。

### 5.1 展示几条

默认展示前 5 条最相关来源。超过 5 条折叠，用户点「查看全部来源」。

排序：

1. 用户批注或高亮过的来源。
2. 被正文引用最多的来源。
3. 置信度高的来源。
4. 时间更新的来源。

### 5.2 展示什么

每条来源展示：

- 文件名。
- 页码/段落/时间戳/图片坐标。
- 原文片段。
- 这条来源支持正文中的哪一句。
- 查看原文按钮。
- 处理状态：正常 / 弱提取 / 失败后人工补充。

不展示内部字段名、素材路径、worker 错误。

证据：FR-A02 AC2/AC10 要求原文引用位置和溯源关系；FR-A02 AC14/AC15 要求图片来源精确到坐标并能高亮原图区域；当前 `entries.metadata_json.sourceRange` 已保存 documentId/originalText 等线索。

## 6. 关联知识

关联知识从 `graph_edges` 来，`wiki_links` 只作为正文内链补充。

### 6.1 展示几条

默认每组展示 5 条，支持展开。

分组：

- 前置知识。
- 相关概念。
- 例题/题目。
- 解题方法。
- 易混概念。
- 批注。

### 6.2 展示形式

用列表卡片，不用复杂动画：

```text
前置知识
- 古典概型：理解条件概率前要先掌握样本空间和事件概率。
- 事件独立性：容易和条件概率混淆。

相关题目
- 已知 P(A), P(B|A) 求 P(A∩B)
```

每条卡片显示：标题、关系类型、一句摘要、关系依据：用户确认 / AI 推断 / 同材料共现。

证据：FR-P03 AC4 定义学科关系类型；当前 DB `graph_edges=6256` 且学科相关边 5295 条，足以支撑关联知识；`wiki_links=0` 不应阻断展示。

## 7. 批注

批注分两类。

### 7.1 原材料批注

用户在 PDF/图片/音视频转写上写的批注。

展示位置：来源材料区和批注区。

作用：说明用户当时为什么标注这段材料。它锚定原文位置。

### 7.2 Wiki 批注

用户在 Wiki 页面正文上写的批注。

展示位置：正文旁或页面底部批注区。

作用：说明用户对聚合知识的理解、质疑或补充。它锚定 Wiki 段落。

两类批注都要进入 Entry/Graph，不能只做前端便签。

证据：FR-A02 定义批注/强调层；FR-P03 AC4 有 annotated_with；DB entry_type 已出现 annotation。

## 8. 图谱位置

每篇 Wiki 页面需要一个 mini graph，但只展示 1 跳关系。

默认显示：

- 当前页面在中心。
- 左侧是前置知识。
- 右侧是应用/题目/方法。
- 下方是易混概念和批注。

点击「打开完整图谱」进入 `/graph?focus=<pageId>`。

如果图谱数据为空，显示：

> 这页还没有建立关系。处理更多材料或手动确认关系后会出现在这里。

证据：FR-G03 要求图谱可视化，FR-W16 要求图谱导航闭环；当前 Wiki 详情已有「在图谱中查看关系」链接。

## 9. 学习状态

KIVO 页面不展示学习进度、掌握度、正在阅读、已学会。

如果接入 exam-sprint，只展示外部应用传来的简短状态入口，例如：

> 在 exam-sprint 中有 3 道题关联这页。

不在 KIVO 内部做掌握度条和学习计划。

证据：spec 设计硬约束明确写明 KIVO Web 不是学习 APP，不做学习进度、学习管线可视化、掌握度等场景化语言。

## 10. 版本和编辑历史

需要，但不能抢主屏。

默认只显示：

- 最近更新时间。
- 最近编辑者：AI / 用户。
- 「查看历史」链接。

历史页展示：版本号、时间、变更摘要、还原按钮。

证据：`wiki_page_versions` 表存在；FR-P07 AC2/AC3/AC5 提到版本历史和还原。

## 11. 置信度和争议性

不展示一个冷冰冰的 confidence 数字。展示用户能理解的可信线索：

- 来源充足：来自 3 份材料、8 处原文。
- 只有一份材料支持：只来自 1 份材料。
- AI 未确认：系统自动生成，用户未编辑。
- 用户确认：用户编辑或确认过。
- 存在争议：用户标记不同意或来源之间冲突。

用户操作：

- 标记可信。
- 标记不同意。
- 补充来源。
- 删除错误内容。

证据：FR-W03 AC5 和 FR-W15 AC4 都要求详情/列表不展示置信度字段；FR-P03 AC5 定义关系 confidence；FR-P07 定义观点分歧和用户干预。

## 12. 空 Wiki / 弱 Wiki 兜底

### 12.1 刚导入材料，还没生成 Wiki

页面显示：

```text
这份材料正在处理
已完成：上传、材料识别
正在进行：抽取知识
下一步：生成 Wiki 页面和关系

你可以：查看材料状态 / 稍后刷新 / 删除材料
```

不显示空正文。

### 12.2 抽取结果太少

页面显示：

```text
这页信息还不够完整
当前只抽到 2 条知识，来源较少。
建议：补充材料 / 手动添加知识 / 重新处理材料。
```

### 12.3 处理失败

页面显示：

```text
这份材料没有生成 Wiki
原因：没有读出文字 / OCR 不可用 / 音频转写失败
你可以：重新处理 / 粘贴文本继续处理 / 删除材料
```

证据：backlog 记录假 done 和空 wiki_page 污染知识库；陌生人走查记录存在「暂未提取出合格知识条目」空壳，影响第一印象。

## 13. 每个 Section 的数据来源

- 标题：`entries.title`，用户编辑后锁定人工版本。
- 摘要：`entries.summary`。
- 正文：`entries.content`，按 Wiki 编译模板渲染。
- 标签：`entries.tags_json`、`entry_type`、`subject_id`。
- 来源材料：`entries.source_json.materialId`、`metadata_json.sourceRange`、`materials`。
- 关联知识：`graph_edges` + `graph_nodes` + `entries`。
- 批注：`entries.entry_type='annotation'` + `graph_edges.annotated_with`。
- 图谱位置：`graph_edges` 1 跳关系。
- 版本历史：`wiki_page_versions`。
- 状态线索：`metadata_json` 中的编译信息、用户编辑记录、争议标记。

## 14. 必选、可选、兜底

必选：

- 标题。
- 摘要。
- 正文。
- 来源材料。
- 关联知识。
- 编辑/删除/标记不同意。
- 图谱入口。

可选：

- mini graph。
- 版本历史完整列表。
- 批注侧栏。
- 全量来源展开。

兜底：

- 没有来源：提示「缺来源」，引导补充材料。
- 无关联：提示「还没有关系」，不显示空白框。
- 无正文：显示处理状态和可操作动作。
- 抽取失败：显示失败原因和人工介入路径。

## 15. 当前实现差距

现有 `web/app/(dashboard)/wiki/[space]/[slug]/page.tsx` 已有正文、来源溯源、历史版本、关联链接的骨架。

缺口：

- 来源区还依赖 sourcePages，不足以展示材料页码/片段/置信线索。
- 关联区当前默认看 `wiki_links`，而 DB 里 `wiki_links=0`，应改为优先展示 `graph_edges`。
- 页面标题仍可能是文件名。
- 批注没有成体系。
- 空 Wiki/弱 Wiki 的状态还不够诚实。
- 学习状态应保持不展示，避免越界到 exam-sprint。

## 16. 最小可交付模板

第一版 Wiki 页面按这个顺序做：

1. 标题 + 摘要 + 状态线索。
2. 正文。
3. 来源材料，至少 3 条，能点回原文。
4. 关联知识，来自 graph_edges，按类型分组。
5. 批注。
6. 图谱入口。
7. 版本历史入口。
8. 空状态/失败状态。

这套模板跑通后，再做更复杂的视觉和交互。
