# KIVO - 用户可见层需求规格(补充)

OpenClaw(pm-01 子Agent)| 2026-04-20

---

## 定位

本文档补充 KIVO 的用户侧需求。现有 spec(product-requirements.md)覆盖了引擎内部的 17 个 FR(提取、存储、检索、冲突、调研、注入、规则分发),但没有定义用户通过 Web 界面能看到什么、能操作什么。

目标用户:产品负责人 / 老板--不看代码,不用 CLI,通过浏览器了解和管控自己的知识库。

设计原则:
- 只暴露对用户决策有价值的信息,不把内部机制全部搬上界面
- MVP 聚焦"看得见、搜得到、能干预",不做管理后台
- 界面是引擎能力的窗口,不是独立产品;所有数据来自引擎已有接口

---

## 功能需求

### 域 G:知识浏览与搜索(Knowledge Browse & Search)

用户想回答的核心问题:我的知识库里有什么?

#### FR-G01:知识库总览仪表盘

用户打开 KIVO 首页,一眼看到知识库的整体状态。

- AC1:展示知识条目总数,按六种知识类型(fact / methodology / decision / experience / intent / meta)分布。
- AC2:展示知识条目按状态分布(active / pending / deprecated)。
- AC3:展示最近 7 天的知识变化趋势(新增、更新、废弃数量)。
- AC4:展示知识库健康指标:待确认条目数、未解决冲突数、过期待清理数。

**优先级:MVP**

#### FR-G02:知识条目列表与筛选

用户能浏览知识条目,按条件筛选定位。

- AC1:知识条目列表展示:标题/摘要、类型、状态、来源、最后更新时间。
- AC2:支持按知识类型、状态、来源、时间范围筛选。
- AC3:列表支持排序(按时间、按相关度)。
- AC4:每页展示条目数可配置,支持分页。

**优先级:MVP**

#### FR-G03:知识全文搜索

用户用自然语言搜索知识库内容。

- AC1:搜索框支持自然语言输入,返回语义相关的知识条目。
- AC2:搜索结果高亮匹配片段,展示相关度评分。
- AC3:搜索结果可按知识类型、状态二次筛选。

**优先级:MVP**

#### FR-G04:知识条目详情

用户点击某条知识,看到完整信息。

- AC1:展示知识条目的完整内容、类型、来源引用、创建/更新时间、版本号。
- AC2:展示该条目的关联知识(补充、替代、冲突、依赖)。
- AC3:展示该条目的版本历史(何时创建、何时被更新、被谁/什么触发)。

**优先级:MVP**

---

### 域 H:知识库动态(Knowledge Activity)

用户想回答的核心问题:知识库最近在自动做什么?

#### FR-H01:活动流(Activity Feed)

用户看到知识库的实时动态,了解系统在做什么。

- AC1:按时间倒序展示知识库事件:新知识入库、冲突检测与解决、知识过期、调研完成、规则变更。
- AC2:每条事件包含:事件类型、摘要、时间、关联的知识条目链接。
- AC3:支持按事件类型筛选。

**优先级:MVP**

#### FR-H02:冲突与待决事项

用户看到需要人工介入的事项。

- AC1:展示当前所有"人工裁决"状态的冲突,包含冲突双方内容对比。
- AC2:用户可在界面上做出裁决:选择保留哪一方,或标记为"两者共存不矛盾"。
- AC3:展示待确认(pending)的知识条目,用户可确认入库或拒绝。

**优先级:MVP**

---

### 域 I:调研可见性(Research Visibility)

用户想回答的核心问题:系统在自动调研什么?进展如何?

#### FR-I01:调研队列展示

用户看到当前的调研任务列表和状态。

- AC1:展示调研任务列表:目标、状态(queued / running / completed / failed)、优先级、创建时间。
- AC2:已完成的调研任务展示结果摘要和入库的知识条目数。
- AC3:失败的调研任务展示失败原因。

**优先级:MVP**

#### FR-I02:缺口报告展示

用户看到系统发现的知识盲区。

- AC1:展示最近的缺口分析报告:盲区主题、影响面(关联了多少次未命中查询)、建议的调研方向。
- AC2:用户可查看历史缺口报告,了解盲区填补进度。

**优先级:MVP**

---

### 域 J:用户操作(User Actions)

用户想做的事:在关键节点能干预,而不只是看。

#### FR-J01:手动触发调研

用户主动指定一个主题让系统去调研。

- AC1:用户输入调研主题和范围描述,系统生成调研任务并加入队列。
- AC2:用户可设置调研优先级(高/中/低)。
- AC3:调研完成后用户收到通知(界面内通知)。

**优先级:MVP**

#### FR-J02:知识标记与干预

用户对知识条目做人工干预。

- AC1:用户可将任意 active 条目标记为"已过时"(触发 deprecated 流程)。
- AC2:用户可将 pending 条目确认为 active 或拒绝(归档)。
- AC3:用户可编辑知识条目的摘要内容(生成新版本,旧版本保留)。

**优先级:MVP**

#### FR-J03:调研任务管理

用户对调研队列做干预。

- AC1:用户可取消排队中的调研任务。
- AC2:用户可调整调研任务优先级。
- AC3:用户可暂停/恢复自动调研(全局开关)。

**优先级:MVP**

---

### 域 K:洞察与统计(Insights & Analytics)

用户想回答的核心问题:知识库对我有多大价值?哪里还薄弱?

#### FR-K01:知识覆盖度地图

用户看到知识库在各领域的覆盖情况。

- AC1:按领域/主题聚类展示知识分布,标注每个领域的条目数量。
- AC2:标注薄弱领域(条目少、查询多但命中低的领域)。
- AC3:覆盖度数据支持按时间对比(本周 vs 上周)。

**优先级:Future**

#### FR-K02:知识利用率统计

用户看到知识被 Agent 使用的情况。

- AC1:展示最近 N 天内被检索命中次数最多的知识条目 Top-N。
- AC2:展示从未被命中的"沉睡"知识条目列表。
- AC3:展示检索未命中的高频查询(Agent 问了但知识库没答上的)。

**优先级:Future**

#### FR-K03:规则分发状态

用户看到规则的分发和生效情况。

"规则"在本界面的含义:规则是 Agent 执行任务时必须遵守的行为约束(编码规范、流程原则、安全策略等),由引擎层的 Rule Entry 结构化管理。规则与知识条目的区别:规则是"必须/禁止/应当"的执行约束,知识条目是"是什么/怎么做/为什么"的认知资产。用户在本界面只能查看规则分发状态,规则的创建和编辑通过引擎层治理文件提取流程完成(参见引擎 spec FR-A03、FR-F01)。

- AC1:展示当前生效的规则列表,每条规则的订阅者数量和最近分发时间。
- AC2:展示分发失败的规则和失败原因。

**优先级:Future**

### 域 L:系统字典(System Glossary)

用户想回答的核心问题:我的专有术语,Agent 是否真的理解?

#### FR-L01：系统字典管理

用户在 Web 界面维护一份专有名词词表，词表中的术语定义对所有 Agent 强制生效——Agent 处理任务时遇到已定义术语，必须按词表定义理解，不再向用户发起澄清。

核心目的：纠偏。消除 Agent 对项目/团队/业务专有术语的误解和反复确认。

字典与知识条目的关系：
- 系统字典是索引层，管“术语映射”——把同义词、别名、缩写统一指向标准定义，确保 Agent 用词一致。
- 知识条目是内容层，管“事实陈述”——记录关于某个主题的具体知识（事实、方法、决策等）。
- 两者不冲突：字典解决“这个词是什么意思”，知识条目解决“关于这个主题我们知道什么”。例如：字典定义“KIVO = Knowledge Iteration & Vibe Orchestration”，知识条目记录“KIVO 采用六种知识类型分类体系”。
- 优先级：当 Agent 遇到已定义术语时，字典定义优先于知识条目中的描述（字典是权威命名层）。

- AC1:用户可在 Web 界面查看系统字典,展示术语名称、定义、适用范围、创建/更新时间。
- AC2:用户可新增术语条目,至少包含:术语名称(必填)、定义说明(必填)、别名/同义词(选填)、适用范围(选填)。
- AC3:用户可编辑和删除已有术语条目。
- AC4:字典支持按名称搜索和按适用范围筛选。
- AC5:Agent 在处理任务时,必须自动参考系统字典;遇到已定义术语时按字典定义理解,不再发起澄清确认。
- AC6:字典变更实时生效--用户修改术语定义后,后续 Agent 任务立即使用新定义。
- AC7:字典必须在系统启动/会话初始化时自动注册到 Agent 可见的上下文入口(如 AGENTS.md、bootstrap 注入、session context),确保 Agent 每次会话都能访问字典,不依赖 Agent 主动记忆去查。
- AC8:意图理解增强模块(FR-E 域)必须主动对字典做术语匹配--用户输入命中字典中已定义术语时,直接按字典定义理解语义,跳过澄清流程。

**优先级:MVP**

## 非功能需求(用户可见层补充)

- NFR-G1:仪表盘首屏加载时间 ≤ 3 秒。
- NFR-G2:搜索结果返回时间 ≤ 2 秒(复用引擎 NFR-4.1)。
- NFR-G3:界面适配桌面浏览器(1280px+),移动端为 Future。
- NFR-G4:所有用户操作(标记、裁决、触发调研)需在 1 秒内给出界面反馈(乐观更新)。

## 与引擎 FR 的映射关系

| 用户可见层 FR | 依赖的引擎 FR | 说明 |
|---|---|---|
| FR-G01 总览 | FR-B01 存储 | 聚合统计来自存储层 |
| FR-G02 列表 | FR-B01 存储 | 列表查询 |
| FR-G03 搜索 | FR-B02 语义检索 | 直接复用检索接口 |
| FR-G04 详情 | FR-B01 + FR-B03 关联 | 条目详情 + 关联展示 |
| FR-H01 活动流 | 全域事件 | 需引擎暴露事件流接口 |
| FR-H02 冲突待决 | FR-C01 冲突检测 | 人工裁决队列 |
| FR-I01 调研队列 | FR-D02 + FR-D03 | 调研任务状态 |
| FR-I02 缺口报告 | FR-D01 缺口检测 | 缺口分析结果 |
| FR-J01 手动调研 | FR-D02 调研生成 | 用户创建调研任务 |
| FR-J02 知识标记 | FR-C02 过期清理 | 用户触发状态变更 |
| FR-J03 调研管理 | FR-D03 调度 | 用户干预调度 |
| FR-K01 覆盖度 | FR-D01 缺口检测 | 缺口数据的可视化 |
| FR-K02 利用率 | FR-B02 检索 | 检索日志聚合 |
| FR-K03 规则状态 | FR-F01-F03 | 规则分发记录 |
| FR-L01 系统字典 | FR-E01 注入 | 字典术语通过注入管线强制生效 |

---

## 优先级归属

- MVP:FR-G01 ~ G04、FR-H01 ~ H02、FR-I01 ~ I02、FR-J01 ~ J03、FR-L01
- Future:FR-K01 ~ K03

MVP 共 12 个 FR,覆盖"看、搜、动态、干预、纠偏"五个核心场景。Future 3 个 FR 聚焦深度分析,依赖引擎侧更多数据积累后再做。
