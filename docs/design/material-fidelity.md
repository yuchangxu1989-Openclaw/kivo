# 材料保真度规约

OpenClaw(ux-01 子Agent)｜2026-05-25

## 结论

大的 PDF 进入 KIVO 后，用户要得到三个确定承诺：原始文件完整保存，解析结果按页可追溯，知识切片不允许成为唯一原文。

一份 500 页 PDF 即使只提取出 80 条知识，系统也必须保留完整 500 页原件。抽取失败的页不能消失，要在材料详情里标成“未识别/待补充”。

证据：`product-requirements.md` 的 FR-A02 AC10 要求原始文档与知识条目保持页码/时间戳溯源；FR-P06 AC5 要求 Wiki 页面展示来源文档页码。当前 `materials.storage_path` 已保存原始文件路径，`web/lib/wiki-materials.ts` 会把上传字节写入 `uploads/wiki-materials`。

## 保真度分层

### 1. 原始文件层

原始 PDF、图片、音频、视频、文本按上传时的字节完整保存。

用户在材料详情里必须能看到：文件名、文件大小、导入时间、处理状态、原文件打开入口、下载入口。

证据：当前 `materials` 表已有 `file_name / mime_type / file_size / storage_path / status / error_message` 字段；`persistUploadedMaterial()` 用 `fs.writeFile(storagePath, Buffer.from(arrayBuffer))` 保存原始文件。

### 2. 页面/时间轴层

PDF 按页建索引；图片按整张图建索引；音频/视频按时间段建索引。

PDF 的最小定位单位是“页 + 段”。手写扫描页如果 OCR 失败，页仍然保留，只是识别文本为空。

证据：`web/lib/document-parsers.ts` 的 PDF 解析已生成 `sourceRange.page / sourceRange.paragraph / originalText`；FR-A02 AC2 要求保留页码、时间戳、截图坐标。

### 3. 知识切片层

切片只负责给 LLM 提取知识，不能代表原文完整性。

每个切片必须带回原始定位：材料 ID、文件名、页码范围、段落范围、原文片段、识别质量。

证据：当前 `web/lib/queue/pipeline-worker.ts` 插入 entry 时会写 `source_json.reference=material://<id>` 和 `metadata.sourceRange.documentId/originalText`，但 `pageNumber` 当前写成 `null`，页码保真还没完整落地。

## 数字规则

### 单文件大小上限

材料库入口采用 400MB 上限。

推导：
- 用户明确关心“大 PDF 不要被截断”。教材、扫描讲义、带图 PDF 很容易超过 50MB。
- FR-A02 AC8 已写“单文件上限 400MB，大文件流式分片处理”。
- 当前材料库上传接口 `web/app/api/v1/wiki/upload/route.ts` 使用 `MAX_MATERIAL_FILE_SIZE_BYTES = 400 * 1024 * 1024`。
- 当前 Web 导入老接口 `web/app/api/v1/knowledge/import/route.ts` 仍是 50MB，FR-W08 AC1 也写 50MB。这和材料库目标冲突。

执行规约：领域知识材料库按 400MB；旧知识导入入口如果继续存在，只能作为轻量导入，页面必须明确“轻量入口 50MB”。最终产品应统一到材料库 400MB。

### 单 PDF 页数上限

默认承诺 500 页完整保存和可浏览；超过 500 页仍保存原文件，但进入“超长材料模式”。

推导：
- 用户问的是“大的 PDF 进去后不要被截断”，核心是保存和可回看，不是一次性把全部页喂给 LLM。
- 500 页教材按每页 1,000～2,000 字估算，全文 50～100 万字，超出单次 LLM 处理窗口，必须流式处理。
- 500 页以内可以给用户稳定的页码索引、目录、进度；超过后分批解析，不影响原件完整保存。

页面表现：超过 500 页时显示“已保存全文，正在分批解析：已处理 120/800 页”。

### 单切片字数

知识抽取切片目标 2,400 字符，上限 3,200 字符；切片之间保留 150～300 字符重叠。

推导：
- 当前 `pipeline-worker.ts` 的 `CHUNK_TARGET_CHARS` 默认就是 2400。
- 2,400 字符足够覆盖一段定义、例题、证明片段，也不会让 LLM 一次处理太多无关内容。
- 重叠区用于防止公式说明、题干和解法被切断。

当前缺口：现有 `chunkPdfText()` 只按空行聚合，没有页码边界和重叠区；`CHUNKS_PER_MATERIAL_CAP` 默认 12，意味着长 PDF 当前最多处理 12 个切片，会发生抽取层截断。

设计要求：原文不截断，抽取分批，不设永久切片上限。为控成本可以设“本轮处理 12 个切片”，但必须继续排队处理剩余切片，不能把剩余内容静默丢掉。

### 最大保留页数

原始文件不按页数删除。页索引默认保留全部页；缩略图可延迟生成。

保留策略：
- 原始文件：用户删除前永久保留。
- 页面文本层：永久保留，随原文件删除。
- 缩略图/预览图：可缓存，可重建。
- LLM 中间结果：可重建，不作为唯一真相。

证据：当前 DB 没有保留期限字段，`materials.storage_path` 是原文件唯一保存线索。删除动作在材料页会提示“关联的 Wiki 页面也会被删除”，说明原件生命周期应由用户动作控制。

## PDF / 图片 / 手写内容处理

### 可复制文字 PDF

直接提取文本，按页建立目录和段落定位。

用户点击 Wiki 的来源时，打开原 PDF 第 X 页，并高亮对应段落。

### 扫描 PDF

每页先保存为页面记录，再进入 OCR。

OCR 失败的页在结构树里显示“第 42 页：无法识别”。用户可以补文字，或上传外部 OCR 结果。

证据：FR-A02 AC6 要求 PaddleOCR 本地优先、Mathpix 付费增强；当前 DB 中图片材料已有失败记录：`OCR not configured (PaddleOCR not on PATH...)`，说明系统现在能记录 OCR 缺失，但还没有补救 UI。

### 含图表 PDF

图表不能只抽成一句摘要。保真规则是：原图保留，图表 OCR/解析结果作为结构化附属层。

Wiki 中展示图表知识时必须带“查看原图/原页”。

### 手写笔记

手写识别结果必须带质量标签：高、中、低、无法识别。

低质量识别不得自动写成确定知识，只能进入待确认队列。

证据：FR-A02 AC6 要求手写 OCR；FR-A02 的批注/强调层要求手写旁注、高亮、圈画进入中间表示。

## OCR 失败后的用户补救

材料详情里每个失败页提供三个动作：

1. 粘贴这页文字。
2. 上传外部 OCR 结果。
3. 标注“该页无法识别，跳过抽取”。

用户补充后，这一页重新进入抽取队列，原始文件不变，补充文本作为用户校正层保存。

页面文案：
- “第 12 页已保存，但 OCR 失败。”
- “你可以补一段文字，KIVO 会把它和原图一起作为来源。”
- “跳过只影响知识抽取，不会删除原页。”

## 超大材料降级

### 文件超过 400MB

不直接丢弃。页面给出两个选择：

1. 保留到本地材料暂存区，提示用户拆分上传。
2. 如果部署环境支持后台流式上传，进入“超大材料队列”。

需 CEO 拍板：是否把 400MB 提升到 1GB。1GB 会带来上传超时、浏览器内存、备份成本、磁盘清理策略等产品问题。当前 spec 和代码都支持到 400MB，先不改成 1GB。

### 抽取失败

材料仍然是“可查看原文”的状态，抽取状态显示失败原因。

禁止把抽取失败显示成“材料不可用”。用户至少可以打开原文件、下载原文件、补 OCR 文本、重新处理。

证据：当前 `processMaterial()` 对图片 OCR 缺失会写 `skipped` 和失败原因；`pipeline-worker.ts` 对 PDF 解析失败会写 `pipeline_status='failed'`。材料列表已展示状态，但缺少“部分可用”的细分。

### 只处理了前 12 个切片

用户必须看到“已保存全文；知识抽取仅完成 12/83 个切片”。

这类状态叫“部分完成”，不能叫“完成”。

证据：当前 `pipeline-worker.ts` 用 `allChunks.slice(0, CHUNKS_PER_MATERIAL_CAP)`，默认 cap=12；但 `markPipelineDone()` 会把材料标成 done。这个行为和用户“大 PDF 不要被截断”的要求冲突。

## 当前实现差距

1. 材料库上传接口支持 400MB，旧知识导入接口仍限制 50MB。
2. 原始文件已保存到 `uploads/wiki-materials`，但页面缺少稳定的全文打开/下载规约。
3. PDF 解析已有页码能力，后台抽取写 entry 时页码没有落入最终 entry。
4. 后台 PDF 管线有 `CHUNKS_PER_MATERIAL_CAP=12`，长文档抽取层会截断。
5. 图片 OCR 缺失会被诚实标记为 skipped，但没有用户补救入口。
6. 当前材料记录里大量 `classification_status=failed`，说明分类/解析失败会阻断后续管线，材料详情必须给用户下一步动作。

## 验收口径

用户上传 500 页 PDF 后，应能完成以下动作：

1. 在材料库看到完整文件大小和总页数。
2. 打开全文预览，从第 1 页翻到第 500 页。
3. 下载原始 PDF。
4. 看到抽取进度，例如“已处理 180/500 页”。
5. 点击任意知识条目，回到原 PDF 页码和段落。
6. OCR 失败页有明确标记和补救按钮。
7. 部分抽取完成时，系统不能说“全文处理完成”。
