# 自动归类与树形结构规约

OpenClaw(ux-01 子Agent)｜2026-05-25

## 结论

导入材料后必须自动归类到领域树下。高置信度自动挂载，低置信度进入待确认，不允许静默丢到错误领域。

领域树统一用一套实体：领域节点。现有 `subject_nodes` 和 Wiki 目录概念不能继续分裂成两套真相。

证据：FR-P04 AC1 要求知识提取管线根据语义自动匹配领域知识库；AC2 要求低置信度归入默认未分类；AC4 要求必须用向量语义检索，禁止关键词规则。当前 `subject_nodes` 有 5 个 auto 节点，DB 没有 `wiki_directory` 表，说明目录层和领域层已经不一致。

## 自动归类输入

归类不只看文件名。

推荐输入优先级：

1. 用户显式选择的目标领域。
2. OCR/文本抽取出的正文摘要。
3. PDF 目录、章节标题、图片识别类型、音频转写摘要。
4. 文件名。
5. 已有领域树名称、别名、范围定义、历史材料。

文件名只能作为辅助。`probability.pdf` 可以提示概率论，但不能替代正文判断。

证据：当前旧分类器 `web/lib/materials/classifier.ts` 把 `title/source_ref/asset_kind` 和 L0/L1 学科域名交给 LLM，信息不足；新 `web/lib/classify/subject_classifier.ts` 已设计为“内容 embedding → subject_nodes 向量候选 → LLM rerank”，更接近 FR-P04。

## 判定规则

### 第一步：向量召回

把材料正文摘要 embedding 后，与领域节点名称、别名、范围定义做向量相似度，取 Top 10 候选。

证据：`subject_classifier.ts` 已使用 `embedQuery` 和 `embedBatch`，并合并 `subject_aliases.alias_embedding`。

### 第二步：LLM 复核

LLM 只在候选领域里判断主领域、可能副领域、置信度、理由。

LLM 不允许凭空发明 node id。若建议新领域，只能进入“建议新建领域”。

证据：`materials/classifier.ts` 已有“NEVER invent subjectNodeId values”校验；`subject_classifier.ts` 有候选向量和新领域建议逻辑。

### 第三步：阈值分流

置信度 ≥ 0.70：自动归类。  
0.50 ≤ 置信度 < 0.70：进入待确认，展示推荐领域和理由。  
置信度 < 0.50：进入未分类。

推导：
- 0.70 已是当前代码阈值：`KIVO_CLASSIFY_THRESHOLD || 0.7`。
- 低于 0.70 时错挂风险大，错误领域会污染后续检索和 Wiki 聚合。
- 0.50 以下说明向量候选和 LLM 判断都弱，默认未分类比误挂更安全。

## 树形结构

领域树默认三层：领域 → 主题 → 知识点。

示例：数学 → 概率论 → 条件概率。

支持继续向下扩展，但 UI 默认只展开三层。超过三层时折叠，避免树变成文件系统迷宫。

证据：FR-P01 AC5 给出的例子就是“数学 → 概率论 → 条件概率 → 贝叶斯定理”，说明多级目录是 spec 目标。

## 用户能否改树

用户可以改。

允许动作：

- 新建领域
- 新建子主题
- 重命名节点
- 移动节点
- 合并节点
- 把材料或知识从一个节点迁到另一个节点

迁移规则：

1. 移动节点时，子节点一起移动。
2. 材料主领域迁移后，材料产出的知识默认跟随迁移。
3. 单条知识有多标签时，只迁移被选中的主领域标签，副领域标签保留。
4. 合并节点后旧节点保留 alias，旧链接不失效。

证据：当前 `subject_nodes` schema 有 `merged_into` 相关读取逻辑，`subject_aliases` 也参与分类召回，说明合并/别名机制已有基础。

## 统一实体方案

统一为 `domain_node` 概念，落库可以继续沿用 `subject_nodes`，但产品语义改为“领域节点”。

一个领域节点承担三件事：

1. 领域知识库里的目录节点。
2. 自动归类的目标节点。
3. Wiki 页面和材料的挂载位置。

当前 `subject_nodes` 内容：

- 概率论与数理统计，父节点为一个现有节点，origin=auto，level=1。
- 高等数学，父节点为同一个现有节点，origin=auto，level=1。
- 认知科学，origin=auto，level=0。
- 生物信息学，origin=auto，level=0。
- 通用学习资料，origin=auto，level=0。

证据：实测 `SELECT id,name,parent_id,tree_kind,origin,confidence,level,status FROM subject_nodes;` 返回 5 行；查询 `wiki_directory` 报 `no such table: wiki_directory`。

设计判断：当前 `subject_nodes` 已存在，`wiki_directory` 表不存在。后续应把 Wiki 目录统一映射到 `subject_nodes/domain_node`，不要再建第二套树。

## 多领域归属

材料支持“主领域 + 副领域”。

规则：

- 主领域用于默认展示、统计、Wiki 聚合。
- 副领域用于检索过滤、图谱关联、跨领域入口。
- 副领域需要显示置信度，不自动覆盖主领域。

示例：一本概率论 PDF 同时包含统计推断章节。

主领域：概率论与数理统计，置信度 0.86。  
副领域：数据分析，置信度 0.62，需要用户确认。  
副领域：高等数学，置信度 0.41，不展示为推荐。

证据：FR-P01 AC6 要求知识条目支持多标签；FR-P04 当前只写“最相关领域知识库”，没有明确材料级副领域。材料级多归属需要 CEO 拍板写入 spec。

## 知识点挂载

材料先归到领域/主题；抽取出的原子知识可以挂到更细节点。

流程：

1. 材料归类到“概率论”。
2. 切片抽取出“条件概率”“贝叶斯公式”。
3. 系统建议创建/挂载到“概率论 → 条件概率”。
4. Wiki 聚合页面跟随知识点节点生成。

材料归类和知识点归类不能混成一个动作。整本书可能归概率论，某一章可能归统计推断。

## 空状态

第一次导入、领域树为空时，页面给两个选择：

1. 先上传材料，KIVO 根据内容建议领域树。
2. 先创建一个领域，比如“数学”“公司知识”“产品资料”。

上传完成后，如果系统建议新领域，显示：

“我建议新建领域：概率论与数理统计。依据：正文多次出现随机变量、分布函数、条件概率。”

用户可点“确认创建”或“放入未分类”。

证据：`subject_classifier.ts` 支持 `isNewDomain` 和 `suggestedPath`；当前 `materials` 表有 `suggested_subject_name / classification_reason / classification_confidence` 字段。

## 当前实现差距

1. 当前 DB 中多数最近材料 `classification_status=failed`，说明自动归类链路不稳定。
2. 旧分类器只看标题等元数据，新分类器看正文向量，但后台实际入口并未完全统一。
3. `subject_nodes` 存在，`wiki_directory` 不存在；产品需要统一为一棵领域树。
4. 低置信度队列已有 repository 设计，但用户在材料导入后的一眼确认链路仍不足。
5. 材料级“主领域 + 副领域”没有明确 spec，需要 CEO 拍板。
6. 自动归类成功后如何迁移已有 Wiki 页面和图谱关系，当前没有完整 UI 规约。

## 验收口径

用户上传一份概率论 PDF 后，应看到：

1. 系统自动建议“数学 / 概率论与数理统计”。
2. 置信度和一句人话理由。
3. 高置信度自动挂入树；低置信度进入待确认。
4. 用户可改到其他领域。
5. 改树后，材料、知识、Wiki、图谱入口同步更新。
6. 第一次使用时，没有树也能先上传，系统反向生成领域建议。
