# 批注与材料结构的可见性规约

OpenClaw(ux-01 子Agent)｜2026-05-25

## 结论

批注必须在三个地方一眼可见：材料预览页、知识详情页、检索结果页。

材料结构必须在打开材料后的第一屏出现。100 页 PDF 不能只给一个文件名和一串抽取结果，用户要先看到目录、页码进度、章节层级、批注分布。

证据：用户原话问“关键是能不能一眼找到哪些是批注”“一眼看清楚材料的结构”。FR-A02 把批注/强调层列为 Document AST 中间表示，FR-P02 AC5 把 `annotation` 定义为领域知识条目类型。

## 批注的视觉规则

### 材料预览页

材料预览页左侧是结构树，右侧是原文预览。

批注在原文里用浅黄色底标出；旁边有一个黑色线框小标签，写明批注类型：高亮、评注、不同意、提问。

页面右侧有“批注索引”，按页码列出全部批注：

- 第 3 页：高亮 2 条，评注 1 条
- 第 7 页：提问 1 条
- 第 12 页：不同意 1 条

用户点索引，右侧原文滚到对应页和段落。

视觉约束：白底黑字。批注颜色只作为辅助，文字标签必须存在，不能靠颜色区分。

### 知识详情页

如果某条知识来自用户批注，标题下方显示来源胶囊：

“来自批注 · 第 12 页 · 原材料《xxx.pdf》”

正文中引用的原文块左侧加一条黑色竖线，块头写“用户批注原文”。

如果知识是 LLM 从普通原文提取的，只显示“来自原文”。这两个来源不能混在一起。

证据：当前 `entries.entry_type` 已有 `annotation` 类型，DB 分布显示 `annotation=2`，说明数据层已经出现批注型条目，但页面还没有形成稳定视觉规则。

### 检索结果页

领域 Wiki 检索结果中，来自批注的结果卡片必须显示：

- 类型：批注
- 批注类型：高亮 / 评注 / 不同意 / 提问
- 来源材料和页码
- 命中片段

卡片左上角加“批注”黑白标签；命中的原文片段用浅黄色 mark。

证据：当前 `web/app/(dashboard)/search/page.tsx` 已用 `<mark class="bg-amber-200">` 做命中高亮，但结果类型只来自 `type`，没有展示 `entry_type=annotation` 和批注来源。

## 批注类型

### 高亮

用途：用户觉得重要。

视觉：浅黄色底 + “高亮”标签。

知识影响：提高抽取优先级；不自动改变知识内容。

证据：FR-A02 写明“被高亮/圈画/批注的内容 → 提取优先级更高”。

### 评注

用途：用户补充解释、自己的理解、课堂老师补充。

视觉：原文旁边显示对话气泡图标，展开后看到评注文字。

知识影响：可以生成 `entry_type=annotation` 的条目；必须标明“来自用户评注”。

### 标记不同意

用途：用户认为材料说法有问题，或与其他资料冲突。

视觉：黑色感叹号图标 + “不同意”标签，不能用红色警报式设计吓用户。

知识影响：进入冲突识别，和原材料结论形成 `conflicts` 或图谱冲突边。

证据：FR-P07 要求领域知识冲突处理；FR-P03 AC4 包含 `confusable_with / annotated_with` 等关系类型。

### 提问

用途：用户看不懂、想追问。

视觉：问号图标 + “提问”标签。

知识影响：可以生成待学习/薄弱点条目；关联到被问的原文段落和相关知识点。

证据：FR-A02 写明手写旁注含“不懂”等要标记为薄弱点/易错点。

## 批注与知识的关系

批注有三个归属层级：

1. 原始材料批注：绑定材料 ID + 页码 + 坐标/段落。
2. Wiki 页面批注：绑定 Wiki 页面 ID + 页面段落。
3. 知识条目批注：绑定 entry ID。

默认批注先归属于原始材料。只有当批注本身有知识价值时，才生成 `entry_type=annotation` 的知识条目。

生成 entry 时必须保留：

- `entry_type=annotation`
- `source_json.reference=material://<id>` 或 wiki/page 引用
- `metadata.annotation.kind`
- `metadata.sourceRange`

证据：当前 DB 没有 `annotations` 表；执行 `.schema annotations` 没有返回建表语句。现有实现只能用 `entries.entry_type=annotation` 表达批注知识，缺少“批注对象本身”的独立存储。

## 材料结构的呈现

### PDF

PDF 打开后第一屏结构：

- 文件标题
- 总页数、已解析页数、失败页数
- 自动目录
- 批注分布
- 抽取出的知识条目数

自动目录优先级：

1. PDF 自带书签目录。
2. 解析正文标题，如“第 1 章”“1.2”“定义”“例题”。
3. 没识别到目录时，按每 10 页聚合为页段：1-10、11-20。

100 页 PDF 的目录示例：

- 第 1 章 随机事件与概率（1-18 页）
  - 1.1 样本空间（2-5 页）
  - 1.2 古典概型（6-11 页）
  - 批注 3 条
- 第 2 章 随机变量（19-41 页）
  - 批注 1 条

证据：`document-parsers.ts` 目前能按页和段落生成 `sourceLocation`，但没有抽取 PDF 书签目录，也没有材料结构表。

### 手写笔记 / 图片

图片没有天然章节，结构按“页面/区域/识别块”呈现。

一张手写图的结构：

- 整图
  - 左上：标题/日期
  - 中部：公式推导
  - 右侧：旁注
  - 下方：例题

如果无法识别区域，显示“整页图片，未识别结构”，并给补充文字入口。

证据：FR-A02 AC11 要求图片先做内容分类，再路由到公式、流程图、表格、手写笔记等策略。当前图片材料 DB 记录显示 OCR 未配置时 `slice_count=0 / extract_count=0`。

### 音频

音频结构按时间轴呈现：

- 00:00-03:20 开场/主题
- 03:21-14:10 概念讲解
- 14:11-22:00 例题
- 22:01-25:00 口头强调

批注对应时间戳，不对应页码。

### 视频

视频结构同时有时间轴和关键帧：

- 时间段
- 关键帧缩略图
- OCR 文字
- 音频转写

用户点结构树节点，视频跳到对应时间点。

## 页面布局

材料详情页固定三栏：

左栏：结构树。  
中栏：原文/预览。  
右栏：批注索引和抽取知识。

用户扫一眼要知道三件事：这份材料分几章，哪些页有批注，哪些地方已经产出知识。

## 当前实现差距

1. DB 没有独立 `annotations` 表，无法保存批注对象、类型、坐标和归属。
2. `entries.entry_type=annotation` 已存在少量数据，但页面没有一眼可见的批注标签体系。
3. `ContentRenderer` 只渲染正文、公式、图片和 PDF 来源引用，没有批注标记能力。
4. 搜索页只展示 `type` 和 `status`，没有展示领域条目类型 `entry_type`。
5. 材料页显示文件状态和大小，但没有 PDF 目录、页码结构、批注分布。
6. 当前 `wiki_links=0`、`graph_edges=6256`，关联关系和 Wiki 链接双轨，批注关系也不能稳定落在一个入口上。

## 验收口径

用户打开一份 100 页 PDF 后，应在 5 秒内完成判断：

1. 这份材料有几章。
2. 哪些页有批注。
3. 哪些批注已经变成知识。
4. 哪些页 OCR 失败。
5. 点任意批注能回到原文位置。
6. 点任意结构节点能跳到对应页。
