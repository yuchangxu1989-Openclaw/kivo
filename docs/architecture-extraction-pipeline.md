## 1. 概述与定位

知识提取管线是 KIVO 域 A（Knowledge Extraction）的核心执行路径，负责从用户对话中自动捕获高价值知识。管线服务于两个功能需求：

- **FR-A05（会话历史知识萃取）**：从已归档的 session 文件中批量提取知识，覆盖存量对话。
- **FR-A06（实时知识提取）**：从实时对话中逐轮自动捕获知识，覆盖增量对话。

两条路径共享同一套预处理、聚合和价值判断机制，区别仅在触发方式和输入来源。

## 2. 管线架构总览

```
用户消息流入
     │
     ▼
┌─────────────────────────────────────┐
│  预处理：系统消息过滤               │  ← 唯一允许的预过滤
│  (heartbeat / system prompt / 路由) │
└─────────────────┬───────────────────┘
                  │ 用户消息全量通过
                  ▼
┌─────────────────────────────────────┐
│  聚合：按时间窗口聚合用户消息       │
│  (debounce 控制触发频率)            │
└─────────────────┬───────────────────┘
                  │ 聚合后的完整消息段
                  ▼
┌─────────────────────────────────────┐
│  LLM 语义判断：是否包含高价值知识   │  ← 价值判断的唯一决策者
│  (纠偏/决策/规则/事实/经验)         │
└─────────────────┬───────────────────┘
                  │ 结构化知识条目
                  ▼
┌─────────────────────────────────────┐
│  标准知识管线入库                   │
│  (冲突检测 → 合并 → 持久化)        │
└─────────────────────────────────────┘
```

## 3. 预处理阶段

用户消息全量入队，不做基于长度或数量的预过滤。

唯一允许的预过滤是识别并排除纯系统生成内容：

| 过滤对象 | 判定方式 | 原因 |
|----------|----------|------|
| heartbeat 消息 | 消息来源标记为系统内部 | 非用户产出，无知识价值 |
| system prompt injection | role=system 且由框架自动注入 | 框架元数据，非用户意图 |
| 内部路由消息 | 消息通道标记为内部路由 | Agent 间调度信号，非用户对话 |

预处理阶段的输出 = 输入中的全部用户消息，无论消息长度是 2 个字还是 2000 个字，一律通过。

---

## 4. 聚合策略

预处理后的用户消息按时间窗口聚合：

- **实时路径（FR-A06）**：每轮对话结束后，将该轮次内所有用户消息聚合为一个提取单元。通过 EXTRACTION_DEBOUNCE_MS 控制触发频率，防止高频消息导致 LLM 请求风暴。debounce 是频率控制机制，不是内容过滤——它延迟触发时机，但不丢弃任何消息。
- **批量路径（FR-A05）**：按 session 文件为单位，将单个 session 内的全部用户消息聚合后送入提取流程。

聚合后的消息段全量送 LLM 语义判断，不做二次筛选。

---

## 5. 价值判断机制

知识价值的判断由 LLM 独占决策，代码层面不做任何基于字数、字符数或消息条数的判断。

LLM 接收聚合后的完整用户消息段，输出结构化判定：

- 是否包含高价值知识（yes/no）
- 如果包含，提取为哪些知识类型（correction / decision / rule / fact_update / lesson_learned）
- 每条知识的结构化内容、来源片段引用和置信度

当 LLM 判定「无高价值内容」时，该消息段不产出知识条目。这是正常的空结果，不是过滤——决策权在 LLM，不在代码。

## 6. 设计约束

### 6.1 禁止项

以下模式在知识提取管线中被明确禁止：

| 禁止的模式 | 禁止原因 |
|------------|----------|
| MIN_SEGMENT_LEN（最小段落长度） | 用消息长度代理知识价值，逻辑谬误 |
| MIN_MESSAGE_LENGTH（最小消息长度） | 短消息可能是最高价值的纠偏指令 |
| MIN_QUEUE_SIZE（最小队列深度） | 单条消息即可包含关键决策 |
| 任何基于字符数的 if 判断 | 字符数与知识价值无因果关系 |
| 任何基于消息条数的 if 判断 | 条数与知识价值无因果关系 |

### 6.2 根因

用户最高价值的纠偏和决策往往是最短的消息。例如：

- "不对" — 2 个字，否定了整个方案方向
- "用 A 不用 B" — 7 个字，确立了技术选型
- "禁止" — 2 个字，建立了永久约束

如果代码用长度阈值预过滤，这些关键信号会被静默丢弃，导致 Agent 反复犯同一个错误。

### 6.3 设计原则

价值判断是语义问题，不是统计问题。语义问题只能由语义理解能力（LLM）解决，不能用数值阈值近似。

## 7. 允许的过滤机制

以下三种机制在管线中被允许使用，因为它们的目的不是过滤知识价值：

### 7.1 NOISE_KEYWORDS 语义类型过滤

- 目的：排除纯操作性对话（重启服务、查状态、部署命令等）
- 性质：语义类型判断，不是长度判断
- 作用点：预处理阶段，在消息入队前识别操作性对话
- 判定依据：消息内容匹配预定义的操作性关键词模式

### 7.2 SECRET_PATTERNS 脱敏

- 目的：防止密钥、token、密码等敏感信息进入知识库
- 性质：安全需求，不是价值判断
- 作用点：聚合后、送 LLM 前，对消息内容做脱敏替换
- 判定依据：正则匹配已知的密钥格式模式

### 7.3 EXTRACTION_DEBOUNCE_MS 进程防抖

- 目的：控制 LLM 调用频率，避免请求风暴
- 性质：频率控制，不是内容过滤
- 作用点：聚合触发时机
- 行为：延迟触发提取，但不丢弃任何消息——debounce 窗口内的所有消息在触发时全量聚合送出

## 8. 两条提取路径

### 8.1 实时路径（FR-A06）

```
message:received 事件
     │
     ▼
入队（全量，不做长度/条数判断）
     │
     ▼
debounce 窗口等待
     │ 窗口到期
     ▼
聚合窗口内所有用户消息
     │
     ▼
NOISE_KEYWORDS 过滤（语义类型）
SECRET_PATTERNS 脱敏（安全）
     │
     ▼
LLM 语义判断 + 结构化提取
     │
     ▼
质量门禁（FR-N05）
     │
     ▼
标准知识管线入库
```

触发机制：hook 监听 `message:received` 事件，每条用户消息到达时立即入队。debounce 机制控制实际提取触发的频率，但队列中的消息不会因为数量少或长度短而被丢弃。

### 8.2 批量路径（FR-A05）

```
cron 定时触发 / CLI 手动触发
     │
     ▼
扫描未处理的 session JSONL 文件
     │
     ▼
解析 JSONL，提取用户消息（全量）
     │
     ▼
NOISE_KEYWORDS 过滤（语义类型）
SECRET_PATTERNS 脱敏（安全）
     │
     ▼
BGE embedding 生成 → 语义聚类去重
     │
     ▼
LLM 结构化知识提取（含三维标签）
     │
     ▼
标准知识管线入库
```

批量路径额外包含语义聚类去重步骤，因为历史 session 中可能存在跨会话的重复表述。

## 9. 运行时流程

### 9.1 实时提取时序

```
User        Gateway     Hook        Queue       LLM         Pipeline
 │            │           │           │           │            │
 │ 消息        │           │           │           │            │
 ├───────────►│           │           │           │            │
 │            │ event     │           │           │            │
 │            ├──────────►│           │           │            │
 │            │           │ enqueue   │           │            │
 │            │           ├──────────►│           │            │
 │            │           │           │           │            │
 │            │           │  debounce │           │            │
 │            │           │  到期     │           │            │
 │            │           │◄──────────│           │            │
 │            │           │           │           │            │
 │            │           │ 聚合全部消息           │            │
 │            │           ├─────────────────────►│            │
 │            │           │           │  判断+提取 │            │
 │            │           │           │◄──────────│            │
 │            │           │           │ 知识条目   │            │
 │            │           │           ├───────────────────────►│
 │            │           │           │           │   入库     │
 │            │           │           │           │            │
```

关键约束：
- hook 异步执行，不阻塞 Gateway 主流程
- 提取失败只记日志，不影响用户对话
- 整条链路 fail-open

### 9.2 批量提取时序

```
Cron        Scanner     Parser      Filter      LLM         Pipeline
 │            │           │           │           │            │
 │ trigger    │           │           │           │            │
 ├───────────►│           │           │           │            │
 │            │ 读 JSONL  │           │           │            │
 │            ├──────────►│           │           │            │
 │            │           │ 用户消息  │           │            │
 │            │           ├──────────►│           │           │
 │            │           │           │ 过滤+脱敏 │            │
 │            │           │           ├──────────►│            │
 │            │           │           │  提取     │            │
 │            │           │           │◄──────────│            │
 │            │           │           │ 知识条目  │            │
 │            │           │           ├───────────────────────►│
 │            │           │           │           │   入库     │
```

关键约束：
- 增量处理，已处理的 session 不重复提取
- 单次处理量可通过 `--limit` 控制 LLM 成本
- 用户活跃判定基于外部渠道真实消息，不基于 session 文件 mtime
