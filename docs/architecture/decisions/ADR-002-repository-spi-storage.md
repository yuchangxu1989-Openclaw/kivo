# ADR-002：Repository 模式 + SPI 实现存储抽象

OpenClaw（sa-01 子Agent） | 2026-04-19

## 状态

Accepted

## 上下文

KIVO 需要在不同宿主环境下运行，底层存储可能是本地文件、SQLite、PostgreSQL 或向量数据库。上层业务逻辑不应感知存储差异。

## 决策

采用 Repository 模式封装数据访问，通过 SPI（Storage Provider Interface）定义存储后端契约。初期实现 SQLite 后端，后续按需实现其他后端。

## 后果

- 业务逻辑与存储实现完全解耦，更换存储后端只需实现新的 SPI Provider
- 初期使用 SQLite 即可满足万级条目规模需求，零额外基础设施
- SPI 契约为后续向量数据库、PostgreSQL 等后端提供清晰的接入规范
- 需要维护 SPI 接口定义和至少一个参考实现
- 语义检索（Embedding + 余弦相似度）场景需要 SPI 覆盖向量操作语义

## 替代方案

| 方案 | 优势 | 劣势 | 否决理由 |
|------|------|------|----------|
| 直接依赖 SQLite API | 无抽象层开销，代码直接 | 更换存储需要修改所有数据访问代码 | 违反解耦约束（§2.1），宿主环境差异大，绑定特定数据库不可接受 |
| ORM 框架（如 Prisma/TypeORM） | 多数据库支持开箱即用 | 引入重量级依赖，ORM 的查询抽象不覆盖语义检索场景 | 语义检索（Embedding + 余弦相似度）超出 ORM 能力范围，仍需自定义 SPI |
